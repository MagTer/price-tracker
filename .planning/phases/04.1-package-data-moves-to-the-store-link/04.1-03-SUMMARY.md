# Phase 04.1 Plan 03: The Domain Engines Move onto the Computed Unit Price — Summary

The scheduler no longer dies at 06:00. The watch evaluation — which lives in
`PriceScheduler._check_alerts()`, **not** in `notifier.py` — now compares against the unit price
computed from the link's own quantity, guards every comparison with an explicit `is not None`, and
the `units_in_package` back-derivation is deleted rather than ported. The service's read paths
carry a computed kr/unit for the first time, and `get_current_deals` stopped silently collapsing
two pack sizes at one store into a single deal row.

## The row shapes 04.1-05 and 04.1-06 render

### `get_links_for_product(product_id) -> list[dict]`

One row per LINK, latest price point per link, **ordered by the computed kr/unit ascending with
`.nulls_last()`** (verified in the compiled SQL — `NULLIF` guard and `NULLS LAST` both present).
Links that still need an amount sink to the bottom instead of masquerading as free.

```python
{
  "product_store_id": str,        "store_id": str,
  "store_name": str,              "store_slug": str,          "store_url": str,
  "package_size": str | None,     "package_quantity": float | None,
  "scraped_package_quantity": float | None,
  "price_sek": float | None,      "offer_price_sek": float | None,
  "store_unit_price_sek": float | None,   # what the STORE printed (D-05) — never sort on this
  "unit_price_sek": float | None,         # the COMPUTED value (D-03) — this is the sortable one
  "in_stock": bool | None,        "checked_at": datetime | None,
  "needs_amount": bool,           # True iff package_quantity is None (D-02's visible flag)
  "quantity_mismatch": bool,      # derived, self-clearing (D-09) — never persisted
}
```

A link with **no price point yet** still appears (outer join): `price_sek` / `unit_price_sek` /
`checked_at` are `None`, `needs_amount` still speaks.

### `get_price_history(product_id, days) -> list[dict]`

```python
{
  "checked_at": datetime,         "product_store_id": str,
  "store_name": str,              "store_slug": str,
  "package_size": str | None,     "package_quantity": float | None,
  "price_sek": float,             "offer_price_sek": float | None,
  "store_unit_price_sek": float | None,
  "unit_price_sek": float | None,  # NEW — computed from the effective price (offer wins)
  "in_stock": bool,                # NEW
}
```

**`unit_price_sek` and `in_stock` are new keys that have never existed.** `mcp_server/server.py:147`
has read both off these rows since extraction and found neither — so `compare_stores` has printed
`N/A` in its Jämförspris column and `Nej` in its Lager column **for every row, always**. The
`priser` skill has never once seen a comparison price. That is a pre-existing bug this plan's
reshape makes fixable; 04.1-05 closes it. `test_get_price_history_carries_the_keys_mcp_reads` is
the guard.

### `get_current_deals()` — the ranking decision (recorded per the plan's output spec)

**Recency ranking KEPT. kr/unit EXPOSED as data, not adopted as the sort key.** CONTEXT.md left
"how `find_deals` ranks now that unit price is computed" to the planner's discretion; re-ranking it
would be a behavior change to an unrelated feature, so each row now carries `package_size` and a
computed `unit_price_sek` and a consumer that wants to sort on them can. Rows also gained
`product_store_id`.

## What Was Built

### `src/domain/service.py`

- **`create_product()`** — `package_size` / `package_quantity` parameters **dropped**. The docstring
  used to instruct the reader to "create separate products for each size variant", which is exactly
  the model this phase abolishes; it now says a product is the abstract good, `unit` is its
  comparison unit, and package data belongs to the link.
- **`link_product_store()`** — gains `package_size` and `package_quantity`, documenting that a NULL
  quantity is legitimate (D-02) and that the first scrape autofills it.
- **`record_price()`** — `price_data["unit_price_sek"]` → `["store_unit_price_sek"]`; the
  `PricePoint(...)` kwarg follows.
- **`check_price()`** — scrape write-path **#2**: calls `apply_scrape_to_link(product_store,
  extraction_result, product.unit)` before `record_price`, logs the mismatch message at WARNING, and
  returns `package_quantity` + `quantity_mismatch` so the caller can surface the flag. The link is on
  the session `record_price` commits, so the autofill persists with the price point — no second commit.
- **`get_products()`** — rows gain `unit`.

### `src/domain/scheduler.py` — the plan's whole reason for existing

- **`_check_single_product()`** — scrape write-path **#1**: writes
  `store_unit_price_sek=extraction.store_unit_price_sek` (the dropped column is gone) and calls
  `apply_scrape_to_link(product_store, extraction, product_store.product.unit)`. `.product` is
  already `joinedload`ed at `scheduler.py:113`, so this costs no extra query and has no lazy-load hazard.
- **`_check_alerts()`** — `current_unit_price` / `regular_unit_price` are now
  `unit_price_py(price, product_store.package_quantity)`. **The back-derivation is deleted.** It
  divided the offer price *by the scraped unit price* to reconstruct a package size — a divisor that
  04.1-02 made legitimately `None`. Every threshold comparison and the drop-% divisor are guarded
  with explicit `is not None` (and `!= 0`).
  `notifier.send_price_alert(unit_price_sek=...)` keeps its kwarg name; only the value's
  **definition** changed.

**`notifier.py` is byte-identical** (`git diff --exit-code` clean), as the plan requires.

## Why the NULL guard is not decoration

`_check_alerts` runs headless at 06:00. A `None <= Decimal` `TypeError` raised inside it is caught by
the per-product `except Exception` in `_check_due_products`, logged, counted as `checks_failed` — and
the operator is never told the watch stopped working. The old code was only *accidentally* safe, via
an `and extraction.unit_price_sek` truthiness short-circuit.

`test_null_quantity_watch_fires_no_alert_and_does_not_crash` calls `_check_alerts` **directly**, so no
handler can mask the exception. **I mutation-verified it:** removing `and current_unit_price is not
None` makes it fail with exactly
`TypeError: '<=' not supported between instances of 'NoneType' and 'decimal.Decimal'`; the guard was
then restored. A test that cannot fail is worthless, and this one can.

A NULL-quantity link still fires its plain `target_price_sek` and any-offer alerts — the guard does
not disable the paths that never touch a unit price (`test_null_quantity_still_fires_a_plain_target_price_alert`).

## The fifth pair-keyed site

`get_current_deals()` deduped on a Python-side `(product.id, store.id)` tuple, which was single-valued
only because of the constraint D-01 drops. Two links at one store — a 24-pack and an 8-pack, the
phase's own acceptance scenario — silently collapsed into ONE arbitrary deal row. Re-keyed to the
link's own id. `test_deals_two_links_same_store` is the regression guard, and it is the only kind of
check that can catch this: 04.1-04's static gate detects a `select()` constrained on both columns,
and a Python-side tuple key is a structurally different shape it cannot see.

## Deviations from Plan

### 1. [Rule 3 - Blocking] `ruff` does not exist in this project — substituted `python -m compileall`

- **Found during:** pre-empted by the orchestrator's ruling and by 04.1-01 / 04.1-02, which both hit it.
- **Issue:** both `<verify>` blocks end in `poetry run ruff check`. Ruff is not in `pyproject.toml`,
  not in `poetry.lock`, and not on PATH. The planner assumed a lint gate this project has never had.
- **Resolution:** substituted `poetry run python -m compileall -q <files>` (clean on all four files).
  **Did not install it** — package installs are excluded from auto-fix, and `poetry.lock` was not touched.

### 2. [Rule 1 - Plan bug, self-inflicted] The `units_in_package` source assertion also matches a comment

- **Found during:** Task 2 acceptance. The criterion asserts `'units_in_package' not in
  inspect.getsource(_check_alerts)`. My explanatory comment *describing the deleted hack* named it,
  so the assert tripped on prose rather than on code.
- **Resolution:** reworded the comment to describe the deletion without the literal token ("dividing
  the offer price BY the scraped unit price"). The assertion is a source-text gate, so any future
  mention of the identifier — even in a comment — will trip it. Worth knowing for 04.1-08.

### 3. [Observation, not a deviation] The predicted RED window did not materialize

The plan (and 04.1-02's summary) expected `tests/test_api.py` and `tests/test_mcp.py` to be red
during this plan. **The full suite is green: 160 passed, 0 failed.** Those two suites mock the
service layer rather than calling it, so the reshaped row dicts and changed signatures never reach
them. This is not a sign their work is done — 04.1-04/05/06 still own the `admin.py`, `schemas.py`
and `server.py` source changes; those tests simply do not currently *prove* anything about the new
row shapes. **04.1-05 and 04.1-06 should treat their test fixtures as needing update on the merits,
not wait for a red bar to tell them so.**

No other deviations. Every acceptance criterion in both tasks was executed as written (with the ruff
substitution) and passed.
