## User Constraints (from CONTEXT.md)

### Locked Decisions

- **D-01:** Drop `uq_product_store(product_id, store_id)`; make `store_url` globally UNIQUE instead. A link *is* a store page, so the URL is its natural key. The existing constraint permits only one link per product per store, which forbids exactly the thing this phase exists to enable. `(product_id, store_id)` → no constraint. Same URL twice → blocked.
- **D-02:** `package_quantity` stays NULLABLE on the link — a link can be saved before its pack size is known. First successful scrape autofills it (D-07); until then the product page renders the link as `kr/unit — needs amount`.
- **D-03:** Unit price is COMPUTED, never scraped: `price_sek / link.package_quantity`. Stores print unit prices with incompatible definitions (kr/rulle vs kr/pack vs kr/100g). **This phase CHANGES the source of truth for unit price — it is not merely relocating a calculation.**
- **D-04:** Compute on read; do not persist unit price. → **`PricePoint.unit_price_sek` is DROPPED.**
- **D-05:** Add `PricePoint.store_unit_price_sek` — the store's *printed* unit price, as scraped. Displayed beside the computed kr/unit, **never sorted on**.
- **D-06:** The scrape stops being the output (unit price) and becomes the input (quantity).
- **D-07:** Autofill when empty; flag on conflict; NEVER overwrite. The typed value is *intent*; the page is *evidence*.
- **D-08:** Extend the parser contract: `PriceExtractionResult` gains `package_amount: Decimal | None` and `package_unit: str | None` alongside the existing `pack_size: int | None`. Normalize via the same st/liter/kg factors the admin UI already has (`PKG_UNITS`).
- **D-09:** Store the page's reading in `ProductStore.scraped_package_quantity` (nullable). Every check writes it. The UI flags the link whenever it differs from `package_quantity`. No state machine; the flag self-clears.
- **D-10:** Product dialog: name / brand / category / **unit only**. Unit is a dropdown of canonical units (st / liter / kg).
- **D-11:** Link dialog gains the packaging chain — "amount in package" + auto-suggested editable label. No unit selector (the product already knows its unit).
- **D-12:** Product page lists all links with package label, current price, and **kr/unit — sortable**.
- **D-13:** MCP `compare_stores` returns one row per LINK, sorted by kr/unit (store + package + price + kr/unit), replacing today's one-row-per-store table.
- **D-14:** Rewrite `0001_initial.py` in place (single clean migration) rather than stacking a 0002.
- **D-15:** ⚠ MANDATORY consequence of D-14 — the DB reset is a required, named step: `docker compose down -v` → `docker compose up -d postgres` → `alembic upgrade head`. The deployed v0.2.0 DB is already stamped `0001`; without the reset `alembic upgrade head` sees `0001 == head` and applies NOTHING.

## Summary

This phase is **not** a schema migration in the usual sense — the DB is empty and `0001_initial.py` is rewritten in place, so the DDL is the *easy* part. The risk is entirely in the **blast radius of two deletions**: dropping `uq_product_store(product_id, store_id)` and dropping `PricePoint.unit_price_sek`. Both have consumers that neither the seed nor CONTEXT.md enumerated, and both fail *at runtime under exactly the conditions the phase creates* rather than at import time.

The single biggest find: **three code paths address a link by `(product_id, store_id)` and call `.scalar_one_or_none()`** — `PUT /products/{pid}/stores/{sid}/frequency` (`admin.py:596-600`), `DELETE /products/{pid}/stores/{sid}` (`admin.py:683-687`), and the bulk-import link dedupe (`admin.py:1538-1544`). Those queries are only single-valued *because of the constraint being dropped*. The moment a user adds "Lambi 8-pack at Willys" alongside "Lambi 24-pack at Willys" — the canonical acceptance-test scenario — they raise `MultipleResultsFound` → HTTP 500. The routes must be re-keyed to `/product-stores/{product_store_id}`, and the import dedupe must key on `store_url` (now the natural key). This is a required part of MODEL-02, not a follow-up.

Second: the watch-evaluation logic that MODEL-03 says must be rewired **does not live in `notifier.py`** (CONTEXT.md points there). `notifier.py` only *renders* the email. The comparison logic is `PriceScheduler._check_alerts()` (`scheduler.py:249-356`), including a back-derivation hack at `:314-321` (`units_in_package = offer_price / unit_price`) that exists only because the scheduler had no access to a package quantity — that hack now **deletes itself**: quantity is on `product_store`, which `_check_alerts` already receives as a parameter. The NULL-quantity guard belongs there.

Third: unit-price computation. Verified against the installed SQLAlchemy **2.0.49**: a `hybrid_property` with an `.inplace.expression` returning a *correlated scalar subquery* over `product_stores` works standalone in `ORDER BY`/`WHERE` (no join required) and Postgres's default `NULLS LAST` on `ASC` puts "needs amount" links at the bottom for free. `NULLIF(package_quantity, 0)` is the correct division guard. Recommendation below is one shared definition used three ways (Python, SQL-with-join, SQL-standalone) so the "one definition by construction" promise of D-03 is structurally enforced rather than merely intended.

**Primary recommendation:** Add `src/domain/pricing.py` with two tiny functions (`unit_price_py`, `unit_price_expr`), have a `PricePoint.computed_unit_price_sek` hybrid delegate to both, and add a single `apply_scrape_to_link()` domain helper that all **three** scrape write-paths call — then fix the three `(product_id, store_id)` routes before touching anything else. Zero new dependencies are needed.

## Computed Unit Price — the mechanism (MODEL-03)

### Verified behavior

I built a scratch model mirroring `PricePoint`/`ProductStore` against the installed SQLAlchemy 2.0.49 and compiled it with the Postgres dialect. **All of the following is [VERIFIED: executed in this session], not assumed.**

**Recommended shape — `src/domain/pricing.py` (new, ~20 lines):**

```python
"""Single definition of unit price (D-03/D-04): price / package_quantity."""
from decimal import Decimal
from sqlalchemy import ColumnElement, func

def unit_price_py(price: Decimal | None, qty: Decimal | None) -> Decimal | None:
    """Python-side unit price. None when the link has no amount yet (D-02)."""
    if price is None or qty is None or qty == 0:
        return None
    return price / qty                      # do NOT quantize here — round at display

def unit_price_expr(price_col: ColumnElement, qty_col: ColumnElement) -> ColumnElement:
    """SQL-side unit price. NULLIF guards qty = 0; NULL / x is NULL for qty IS NULL."""
    return price_col / func.nullif(qty_col, 0)
```

**Then the hybrid on `PricePoint` (models.py):**

```python
from sqlalchemy import SQLColumnExpression, func, select
from sqlalchemy.ext.hybrid import hybrid_property
from domain.pricing import unit_price_expr, unit_price_py

class PricePoint(Base):
    ...
    @hybrid_property
    def effective_price_sek(self) -> Decimal:
        return self.offer_price_sek if self.offer_price_sek is not None else self.price_sek

    @effective_price_sek.inplace.expression
    @classmethod
    def _effective_price_expr(cls) -> SQLColumnExpression[Decimal]:
        return func.coalesce(cls.offer_price_sek, cls.price_sek)

    @hybrid_property
    def computed_unit_price_sek(self) -> Decimal | None:
        ps = self.product_store                       # ⚠ must be eagerly loaded — see Pitfall 7
        return unit_price_py(self.effective_price_sek, ps.package_quantity if ps else None)

    @computed_unit_price_sek.inplace.expression
    @classmethod
    def _computed_unit_price_expr(cls) -> SQLColumnExpression[Decimal | None]:
        return (
            select(unit_price_expr(cls.effective_price_sek, ProductStore.package_quantity))
            .where(ProductStore.id == cls.product_store_id)
            .correlate(cls)
            .scalar_subquery()
        )
```

**Naming — deliberate:** call it `computed_unit_price_sek`, **not** `unit_price_sek`. If you reuse the old column's name, every consumer you *forgot* to update keeps working silently while now returning a value with a **different definition** (computed vs. store-printed). With a new name, a missed reference is an `AttributeError` — loud, at the first test run. This is worth the extra typing.

### Compiled SQL (verified output, Postgres dialect)

Standalone `ORDER BY` — no join needed, correlation is automatic:

```sql
SELECT price_points.id, (SELECT coalesce(price_points.offer_price_sek, price_points.price_sek)
                         / CAST(nullif(product_stores.package_quantity, 0) AS NUMERIC)
                         FROM product_stores
                         WHERE product_stores.id = price_points.product_store_id) AS anon_1
FROM price_points
ORDER BY (SELECT ... ) ASC NULLS LAST
```

Python side, same code path:

| Case | `package_quantity` | price | `computed_unit_price_sek` |
|------|--------------------|-------|---------------------------|
| 24-pack | `24` | 139.90 | `5.8291666…` (Decimal, unrounded) |
| needs amount (D-02) | `None` | 59.90 | `None` — no crash |
| on offer | `24` | 59.90 → offer 39.90 | `1.6625` (offer wins via `coalesce`) |

### Three facts the planner should design around

1. **`ORDER BY … ASC` puts NULL last by Postgres default** (`NULLS LAST` is the default for `ASC`) [CITED: postgresql.org/docs/current/queries-order.html]. Links that "need an amount" therefore sink to the bottom of a cheapest-first sort with no extra work. Write `.nulls_last()` explicitly anyway — it documents the intent and survives a later flip to `DESC`.
2. **In a query that *already joins* `product_stores`, the hybrid still emits its own correlated subquery** — it does not reuse the join (verified). Functionally correct, but it re-looks-up the same row. In the queries that already join (`admin.py:199`, `:370`, `:742`, `service.py:135`, `:181`), prefer calling `unit_price_expr(PricePoint.effective_price_sek, ProductStore.package_quantity)` inline and `.label("unit_price_sek")` it. Same definition, one scan. Use the hybrid where no join exists.
3. **Do not quantize inside the computation.** `139.90 / 24` = `5.829166…`. Rounding belongs at the presentation boundary (`round(x, 2)` / `Decimal.quantize(Decimal("0.01"), ROUND_HALF_UP)` in the dict-builders and Jinja/JS). Rounding early and then comparing is how two "equal" unit prices start disagreeing.

## The Unique-Constraint Swap (MODEL-02)

**Postgres B-tree limit is ~2704 bytes per index entry** (⅓ of an 8 kB page) [CITED: postgresql.org/docs/current/btree.html — "values larger than about one-third of a page cannot be indexed"]. `store_url` is `String(512)`; worst case 512 chars × 4 bytes (UTF-8) = 2048 bytes < 2704. **A unique index on `store_url` as-is is safe** — no truncation, no hash index, no `md5()` expression index needed. [VERIFIED: arithmetic against the documented limit]

**Declare it explicitly, not via `unique=True`, to stay drift-free.** STATE.md (Plan 01-03) records that this project already tripped `alembic check` on a *redundant* `UniqueConstraint` — the lesson is that the migration DDL and the ORM metadata must produce byte-identical constraint sets. Prescription:

```python
# models.py — ProductStore
__table_args__ = (UniqueConstraint("store_url", name="uq_product_stores_store_url"),)
#  ^ the old (UniqueConstraint("product_id", "store_id", name="uq_product_store"),) is deleted

# alembic/versions/0001_initial.py — product_stores create_table()
sa.UniqueConstraint("store_url", name="uq_product_stores_store_url"),
#  ^ replaces sa.UniqueConstraint("product_id", "store_id", name="uq_product_store")
```

Matching *names* on both sides is what keeps `alembic check` silent. (Contrast: `stores.slug` uses `unique=True, index=True` → a unique **index** (`ix_stores_slug`), not a constraint. Don't mix the two styles for `store_url`.)

**API consequence — a 409, not a 500.** The unique violation is now a *reachable user action* ("paste a URL that's already tracked"). `IntegrityError` on `session.commit()` in `link_product_store()` (`service.py:383-385`) currently propagates to admin.py's blanket `except Exception` → **HTTP 500 "Failed to link product to store"**. Catch `sqlalchemy.exc.IntegrityError` and return `409 Conflict` with a message naming the existing link. (ASVS V5 — meaningful validation errors, no stack leak.)

## Alembic In-Place Rewrite (MODEL-08)

**Mechanics — verified against `alembic/versions/0001_initial.py` and the deployment history.**

The failure mode D-15 names is real and is worth restating precisely because it is *silent*:

- `alembic_version` is a one-row table holding the current revision **id**. The deployed v0.2.0 DB holds `0001_initial`.
- `alembic upgrade head` compares that id to the head revision id. If you rewrite the *body* of `0001_initial.py` but keep `revision = "0001_initial"`, the ids match. Alembic concludes there is nothing to do and **exits 0 with no output**. The app then boots against the *old* schema and fails at the first query that touches `product_stores.package_quantity` — not at migration time.
- There is no checksum. Alembic does **not** hash migration bodies (unlike Flyway). Nothing detects the edit. [VERIFIED: alembic's `alembic_version` table has exactly one column, `version_num`]

**Therefore the reset is a plan task with a verification step, not a footnote:**

```bash
docker compose down -v            # drops the pgdata volume: tables AND alembic_version
docker compose up -d postgres
alembic upgrade head              # runs the rewritten 0001 against an empty DB
alembic current                   # must print 0001_initial (head)
alembic check                     # must print "No new upgrade operations detected."
```

`alembic check` [VERIFIED: alembic 1.18.4 ships the `check` command] compares live DB metadata to `Base.metadata` and exits non-zero on drift. It is the cheapest possible gate that the rewritten migration and the rewritten `models.py` agree — **make it a plan verification step.** Watch for these three drift sources specific to this phase:

| Change | Drift trap |
|--------|-----------|
| Drop `uq_product_store`, add `uq_product_stores_store_url` | Name mismatch between model `__table_args__` and migration → drift |
| `PricePoint.store_unit_price_sek` added | Must be `sa.Numeric(10, 2), nullable=True` in *both* places (matching `unit_price_sek`'s old spec) |
| `ProductStore.scraped_package_quantity` added | Same — `Numeric(10, 2)`, nullable |

The docstring at the top of `0001_initial.py` (lines 7-17) describes the squash lineage. Update it to note the 04.1 reshape, or the next reader will believe the file is still a faithful squash of the source-platform migrations.

## Parser & Extractor Contract (MODEL-04, MODEL-05)

### There are THREE producers of a "unit price", not one

The seed and CONTEXT both talk about "the parser". In fact `PriceExtractionResult.unit_price_sek` is populated from three places:

| Producer | Line | What it yields | After this phase |
|----------|------|----------------|------------------|
| LLM prompt (`parser.py`) | `:218-221`, `:261` | The page's printed unit price, **or a computed fallback** `price / pack_size` (`:264-266`) | → `store_unit_price_sek` (D-05). **The fallback at :264-266 must be DELETED** — see below |
| `WillysAPIExtractor` | `willys_api.py:66-99` | `comparePrice` — literally the store's printed compare price ("33,29 kr/kg") | → `store_unit_price_sek` verbatim. Already exactly D-05's semantics |
| `JSONLDExtractor` | `jsonld.py:70` | always `None` | unchanged |

**The `price / pack_size` fallback at `parser.py:264-266` is now actively harmful and must be removed.** `store_unit_price_sek` means *"what the store claims"* (D-05). If the parser silently substitutes its own computation when the page prints nothing, the "Store says" column becomes a mix of store claims and our own arithmetic — destroying the one signal it exists to provide (spotting the Coop `kr/100g` mismatch). Let it be `None`. The computed column already covers that case.

### Renaming the dataclass field

`PriceExtractionResult.unit_price_sek` should be renamed **`store_unit_price_sek`** so the result type says what it means. This is a 1:1 rename across 3 producers + 3 consumers (`scheduler.py:235`, `service.py:516-518`, `admin.py:850-851`) + ~15 test constructions. Mechanical, and it makes every remaining `unit_price_sek` in the codebase a compile-time-visible relic.

### The extended contract (D-08)

```python
@dataclass

    store_unit_price_sek: Decimal | None      # renamed from unit_price_sek (D-05)
    offer_price_sek: Decimal | None

    offer_details: str | None

```

Prompt additions (`parser.py:_build_prompt`, after the existing `pack_size` bullet):

```
- "package_amount": The numeric amount of product in the package, as printed
  (e.g. 0.5 for "0,5 l", 400 for "400 g", 24 for "24-pack"). Null if not stated.
- "package_unit": The unit that amount is expressed in — one of "st", "ml", "l",
  "g", "kg". Null if not stated.
```

And the existing `"unit_price"` bullet must be **rewritten** — today it *instructs the model to compute* (`"If the page shows an irrelevant unit price (like kr/kg for toilet paper), calculate it yourself"`). That instruction now contradicts D-05. Replace with: *"the comparison price exactly as printed on the page (jämförpris), including when its unit differs from the pack unit. Do NOT compute it. Null if the page prints none."*

Also: a scraped `package_unit` whose canonical form ≠ `product.unit` (page says grams, product's unit is `st`) must yield `scraped_package_quantity = None` + a log line — **not** a garbage number. That is a conflict, not evidence.

## Common Pitfalls

### Pitfall 1 — Three endpoints treat `(product_id, store_id)` as a unique key (BLOCKER)

**What goes wrong:** `scalar_one_or_none()` raises `sqlalchemy.exc.MultipleResultsFound` the moment a product has two links at the same store — *the exact scenario this phase exists to enable*. It surfaces as HTTP 500 from the blanket `except Exception` handlers.

**Where** [VERIFIED: grep + read]:
- `admin.py:596-600` — `PUT /products/{product_id}/stores/{store_id}/frequency`
- `admin.py:683-687` — `DELETE /products/{product_id}/stores/{store_id}` (unlink)
- `admin.py:1538-1544` — `/import` link dedupe ("Check if link already exists")

**How to avoid:** re-key the two routes to the link's own id — `PUT /product-stores/{product_store_id}/frequency`, `DELETE /product-stores/{product_store_id}` (`POST /check/{product_store_id}` at `:776` already does this correctly and is the pattern to copy). Update `admin.html`'s callers. For `/import`, dedupe on `store_url` — the new natural key — not on the pair.

**Warning sign:** any surviving `select(ProductStore).where(ProductStore.product_id == …, ProductStore.store_id == …)` after this phase is a latent 500.

### Pitfall 2 — The watch logic is in `scheduler.py`, not `notifier.py`

**What goes wrong:** CONTEXT.md and REQUIREMENTS both say "the notifier's watch evaluation". `notifier.py` only *formats HTML* (`:117-162` renders a `kr/enhet` row if given a value). The actual comparisons — `watch.unit_price_target_sek`, `unit_price_drop_threshold_percent` — live in `PriceScheduler._check_alerts()` (`scheduler.py:249-356`). A planner who edits only `notifier.py` will ship a scheduler that references a dropped column and crashes on the first scheduled check (in production, at 06:00, unattended).

**Two specifics:**
1. `scheduler.py:314-321` back-derives `units_in_package = offer_price / unit_price` to reconstruct a regular unit price. With quantity on the link this whole block **collapses** to `unit_price_py(extraction.price_sek, ps.package_quantity)` vs `unit_price_py(extraction.offer_price_sek, ps.package_quantity)`. Delete the hack; don't port it.
2. `_check_alerts` already receives `product_store` as a parameter, and `scheduler.py:113-115` `joinedload`s `.product` and `.store` — so `product_store.package_quantity` is available **with no extra query and no lazy-load hazard**. Good news; use it.

**NULL-quantity guard (MODEL-03, explicit):** when `package_quantity is None`, both unit-price checks must be *skipped*, not evaluated. `if watch.unit_price_target_sek and (u := unit_price_py(...)) is not None and u <= watch.unit_price_target_sek`. Today's code is accidentally safe (`and extraction.unit_price_sek` short-circuits); the rewrite must preserve that property deliberately. A `None <= Decimal` comparison raises `TypeError` inside the scheduler loop → the per-product `except Exception` at `scheduler.py:157-160` swallows it as `checks_failed`, so the bug would be **silent**. Test it.

### Pitfall 3 — There are THREE scrape write-paths, and D-07 must be in all of them

**What goes wrong:** the autofill/flag/never-overwrite rule gets implemented in the scheduler and forgotten in the manual-check button (or vice versa), so "Check now" in the UI records a price without ever filling `scraped_package_quantity`.

**The three:**
1. `PriceScheduler._check_single_product()` — `scheduler.py:190-247` (creates `PricePoint` directly)
2. `PriceTrackerService.check_price()` — `service.py:441-548` (builds a dict → `record_price()`)
3. `POST /check/{product_store_id}` — `admin.py:776-890` (**duplicates** #2's fetch/parse/record logic inline rather than calling it)

**How to avoid:** one domain helper, called by all three:

```python
# domain/pricing.py (or packaging.py)
def apply_scrape_to_link(link: ProductStore, extraction: PriceExtractionResult,
                         product_unit: str | None) -> str | None:
    """D-07/D-09. Returns a mismatch message, or None. Never overwrites package_quantity."""
    scraped = normalize_to_canonical(extraction, product_unit)   # None if unit conflicts
    link.scraped_package_quantity = scraped                       # always record the reading
    if scraped is None:
        return None
    if link.package_quantity is None:
        link.package_quantity = scraped                           # autofill when empty
        return None
    if link.package_quantity != scraped:
        return f"page says {scraped}, you have {link.package_quantity}"  # flag, don't touch
    return None
```

The flag is *derived* on read (`link.scraped_package_quantity != link.package_quantity`), so it self-clears (D-09) — do not persist a boolean.

### Pitfall 4 — `compare_stores`'s unit-price column has been dead the whole time

**What goes wrong:** you "port" MCP's `unit_price_sek` handling and inherit a bug. `mcp_server/server.py:147` does `row.get("unit_price_sek")` on rows from `service.get_price_history()` — but that method's dict literal (`service.py:144-156`) only emits `checked_at`, `price_sek`, `store_name`, `offer_price_sek`. **`unit_price_sek` and `in_stock` are never present** → `_fmt_price(None)` → the "Jämförspris" column prints `N/A` for every row today, and the "Lager" column always prints `Nej`. [VERIFIED: read both files]

**Implication for D-13:** `get_price_history()` must be reshaped anyway (it is the only data source MCP has). Give it `product_store_id`, `store_name`, `package_size`, `package_quantity`, `price_sek`, `offer_price_sek`, `store_unit_price_sek`, computed `unit_price_sek`, `in_stock` — then `compare_stores` becomes a sort + render, and `check_price`'s `in_stock` bug (`server.py:73`) fixes itself for free. Confirm with the operator that a *silently broken* column has been "working" — MCP output has been under-informing the `priser` skill.

### Pitfall 5 — The unit factors will exist in two languages

`PKG_UNITS` is JS; the scraper normalizer must be Python. Two copies of `{ml: 0.001, g: 0.001}` will drift. Cannot be avoided without a build step (out of scope), so **contain** it: define the Python table once in `domain/`, expose it to the template via the existing server-rendered context (or just accept duplication) and write a test that asserts the Python table matches the literal table in `admin.html` (parse it out with a regex, or assert against a shared JSON fixture). Cheap insurance; a 1000× error (g vs kg) silently makes one link look 1000× cheaper than every other.

### Pitfall 6 — There is no DB-backed test in this repo

**What goes wrong:** the planner writes "add a test that sorts links by kr/unit" and discovers at execution time that `tests/` has **no `conftest.py`** and **no engine** — every test is `AsyncMock`/`MagicMock` [VERIFIED: `ls tests/`, read `test_service.py`, `test_api.py`]. `aiosqlite` is declared in `pyproject.toml` but **unused**, and it cannot be dropped in as-is: the models use `postgresql.UUID(as_uuid=True)` and `JSONB`, which do not compile on SQLite.

**How to avoid — pick one, and say so in the plan:**
- **(a) Compile-only, no DB** *(recommended default)*: assert the hybrid's rendered SQL (`str(stmt.compile(dialect=postgresql.dialect()))` contains `nullif` and `NULLS LAST`) and test `unit_price_py` as a pure function with a table of cases. Zero infra, catches the real bugs (NULL guard, offer precedence, factor errors). Fits the existing mocks-only style.
- **(b) Real Postgres integration test**: `docker compose up -d postgres` is already in the repo; add a `conftest.py` with an `asyncpg` engine + `pytest.mark.integration`, run `alembic upgrade head`, insert Lambi + 3 links, assert the ORDER BY. Highest fidelity — and it is the **only** way to actually prove AC#3 and the `store_url` UNIQUE constraint end-to-end. Costs a session-scoped fixture and a CI decision.
- (c) Making the models SQLite-compatible (`sa.Uuid`, `JSON().with_variant(JSONB, "postgresql")`) — a model change beyond this phase's mandate. **Do not.**

The Nyquist gate below assumes (a) as the per-commit sampling rate and (b) as the phase gate.

### Pitfall 7 — `MissingGreenlet` from the hybrid's Python side

`PricePoint.computed_unit_price_sek` reads `self.product_store`. In async SQLAlchemy, touching an **unloaded** relationship triggers a lazy load in a non-greenlet context → `sqlalchemy.exc.MissingGreenlet: greenlet_spawn has not been called` [CITED: docs.sqlalchemy.org/en/20/errors.html#error-xd2s]. Anywhere the hybrid is used on a `PricePoint` fetched without `joinedload(PricePoint.product_store)` / `selectinload(...)` it will explode at request time (not test time — the mocks won't catch it).

**Rule for the plan:** any query that loads `PricePoint` objects *and* reads the hybrid must add `.options(joinedload(PricePoint.product_store))`. In practice, most call sites (`admin.py:199-260`, `:365-405`, `:740-770`) already select `ProductStore` alongside in the same row tuple — in those, prefer passing `ps.package_quantity` to `unit_price_py()` directly and never touch the hybrid. Reserve the hybrid for SQL-side use.

## Code Examples

### Sorting a product's links by kr/unit (the AC#3 query)

```python
# Latest price per link, ranked cheapest-per-unit first; NULL quantity sinks to the bottom.
from sqlalchemy import select, func
from domain.pricing import unit_price_expr

latest = (
    select(
        PricePoint.product_store_id,
        func.max(PricePoint.checked_at).label("checked_at"),
    )
    .group_by(PricePoint.product_store_id)
    .subquery()
)
up = unit_price_expr(PricePoint.effective_price_sek, ProductStore.package_quantity)
stmt = (
    select(ProductStore, Store, PricePoint, up.label("unit_price_sek"))
    .join(Store, ProductStore.store_id == Store.id)
    .join(latest, latest.c.product_store_id == ProductStore.id)
    .join(
        PricePoint,
        (PricePoint.product_store_id == latest.c.product_store_id)
        & (PricePoint.checked_at == latest.c.checked_at),
    )
    .where(ProductStore.product_id == product_uuid)
    .order_by(up.asc().nulls_last())
)
```

(`admin.py:199-228` already builds a `latest_by_ps` map in Python for the product list; reusing that shape and sorting the resulting dicts in Python is equally acceptable at this scale — the SQL version matters for MCP/`find_deals`, which have no client-side sorter.)

### Watch evaluation with the NULL guard (scheduler.py)

```python
qty = product_store.package_quantity                     # joinedload'd, no lazy hazard
cur_unit = unit_price_py(current_price, qty)             # None when qty is NULL (D-02)
reg_unit = unit_price_py(extraction.price_sek, qty)

if watch.unit_price_target_sek and cur_unit is not None:
    if cur_unit <= watch.unit_price_target_sek:
        should_alert = True

if (watch.unit_price_drop_threshold_percent
        and cur_unit is not None and reg_unit not in (None, 0)
        and extraction.offer_price_sek):
    drop = ((reg_unit - cur_unit) / reg_unit) * 100
    if drop >= watch.unit_price_drop_threshold_percent:
        should_alert = True
        unit_price_drop_percent = float(drop)
```
