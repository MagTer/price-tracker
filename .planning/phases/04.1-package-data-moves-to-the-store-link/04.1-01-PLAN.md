Create `src/domain/pricing.py`. It must import NOTHING from `domain.models` — models will import it, and
the reverse would be a cycle. It may import from `decimal`, `logging`, `typing`, and `sqlalchemy`.

Exact module contents:

1. `PKG_UNITS: dict[str, tuple[str, Decimal]]` — the Python twin of the JS table in `admin.html`, mapping
   entry unit to (canonical unit, factor to canonical): `st` to (`st`, 1), `ml` to (`liter`, 0.001),
   `l` to (`liter`, 1), `g` to (`kg`, 0.001), `kg` to (`kg`, 1). Use `Decimal` factors, never floats.
2. `CANONICAL_UNITS: tuple[str, ...]` — the three values `Product.unit` may hold: `st`, `liter`, `kg`.
3. `MAX_PACKAGE_QUANTITY: Decimal` = `Decimal("99999999.99")` — the upper bound of a `Numeric(10, 2)`
   column. A hallucinated `1e30` from the LLM must be rejected here, not surface as a `DataError` at flush
   inside the scheduler tick.
4. `unit_price_py(price: Decimal | None, qty: Decimal | None) -> Decimal | None` — returns `None` when
   `price` is None, `qty` is None, or `qty` equals zero; otherwise `price / qty`. Do NOT quantize or round
   here: rounding belongs at the presentation boundary, and rounding early then comparing is how two
   "equal" unit prices start disagreeing.
5. `unit_price_expr(price_col, qty_col) -> ColumnElement` — the SQL twin: `price_col` divided by
   `func.nullif(qty_col, 0)`. `nullif` is the divide-by-zero guard; a NULL quantity yields NULL by SQL
   semantics, and Postgres sorts NULLs last on an ascending order-by, so "needs amount" links sink to the
   bottom of a cheapest-first sort for free.
6. `normalize_amount(amount: Decimal | None, entry_unit: str | None, product_unit: str | None) -> Decimal | None`
   — look up `entry_unit` (lowercased, stripped) in `PKG_UNITS`; return None if unknown. Return None if the
   entry unit's canonical form differs from `product_unit` — a page reporting grams for a product whose unit
   is `st` is a CONFLICT, not evidence; log it at INFO and return None rather than a garbage number.
   Multiply by the factor; reject a result that is not strictly positive or that exceeds
   `MAX_PACKAGE_QUANTITY` by returning None; quantize the survivor to two decimal places (the storage
   column's precision).
7. `scraped_quantity_from(extraction: ScrapedPackage, product_unit: str | None) -> Decimal | None` —
   D-06/D-08. Prefer `extraction.package_amount` + `extraction.package_unit` through `normalize_amount`.
   If that yields None and `extraction.pack_size` is set and `product_unit` is `st`, fall back to the
   `pack_size` item count, bounds-checked the same way. Otherwise None. `ScrapedPackage` is a
   `typing.Protocol` declared in this module with three attributes: `package_amount: Decimal | None`,
   `package_unit: str | None`, `pack_size: int | None`. Structural typing keeps this module decoupled from
   `domain.result`; `PriceExtractionResult` will satisfy it once 04.1-02 adds the fields.
8. `apply_scrape_to_link(link: LinkLike, extraction: ScrapedPackage, product_unit: str | None) -> str | None`
   — the D-07/D-09 rule, and the ONLY place it is implemented. In order: compute the scraped quantity via
   `scraped_quantity_from`; assign it to `link.scraped_package_quantity` ALWAYS (record the page's reading
   even when it is None); if it is None, return None; if `link.package_quantity` is None, set it to the
   scraped value (autofill-when-empty) and return None; if the stored quantity differs from the scraped one,
   return a human-readable mismatch message naming both numbers (page value first, stored value second)
   WITHOUT touching `link.package_quantity`; otherwise return None. Never overwrite a non-NULL stored
   quantity — the typed value is intent, the page is evidence, and evidence does not get to silently rewrite
   intent now that kr/unit and the entire history view are computed from that number. `LinkLike` is a second
   Protocol with `package_quantity` and `scraped_package_quantity` attributes.
9. `quantity_mismatch(link: LinkLike) -> bool` — the DERIVED flag (D-09): True only when both quantities are
   non-NULL and unequal. Never persist a boolean; deriving it means the flag self-clears the moment either
   number is corrected.

Create `tests/test_pricing.py` — pure unit tests, no DB and no SQLAlchemy session mocks. Define a small local
dataclass stub satisfying `ScrapedPackage` and one satisfying `LinkLike`, in the builder-helper style of
`tests/test_scheduler.py`. Cover:
- `unit_price_py`: a 24-pack at 139.90 returns an unrounded Decimal; NULL quantity returns None; a zero
  quantity returns None; a NULL price returns None.
- `unit_price_expr`: compile it against the Postgres dialect (`sqlalchemy.dialects.postgresql.dialect()`)
  and assert the rendered SQL contains `nullif`.
- `normalize_amount`: 400 with entry unit `g` and product unit `kg` gives `0.40`; 500 `ml` against `liter`
  gives `0.50`; 24 `st` against `st` gives `24.00`; `g` against a product whose unit is `st` gives None
  (unit conflict); an unknown entry unit gives None; a value above `MAX_PACKAGE_QUANTITY` gives None; zero
  and negative give None.
- `scraped_quantity_from`: the `pack_size` fallback fires only when the product's unit is `st`.
- `apply_scrape_to_link`, the three D-07 cases plus the conflict case:
