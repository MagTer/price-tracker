Edit `src/api/templates/admin.html`.

**Product dialog (D-10, AC#1).** Strip the packaging chain out: remove the `pkg-amount` input and the
`pkg-label` input entirely. Keep a unit field, but reduce it to a plain dropdown of the CANONICAL units only —
`st`, `liter`, `kg` (plus an empty option) — because `Product.unit` now stores exactly one of those three and
nothing else. The ml/g entry helpers do not belong here any more: they exist to normalize an AMOUNT, and the
product no longer has one. `handleCreateProduct`'s POST body loses `package_size` and `package_quantity`, and
sends `unit` as the selected canonical value directly (no factor arithmetic). Update the explanatory comment
block above the field — it currently describes a three-field chain that no longer lives here.

**Link dialog (D-11, AC#2).** Add the packaging chain to the link-store form: an "Amount in package" number
input and a "Quantity label" text input that auto-suggests from the amount and stays editable (the
`pkgLabelDirty` latch is what makes it editable — reuse it verbatim). The link dialog needs **NO unit
selector**: the product already knows its unit, so render the amount field labelled IN that unit (for example
"Amount in package (st)"). `showLinkStoreModal` currently takes only a product id — pass the product's `unit`
too (it is already on every product row in `allProducts`) and use it to label the field, pick the input step
(integer for `st`, 0.01 otherwise, exactly as `PKG_UNITS[...].integer` already drives it), and generate the
label suggestion from the matching `PKG_UNITS` entry.

Entry helpers may remain as a convenience where they normalize INTO the product's canonical unit — a product
whose unit is `kg` may accept an amount typed in `g`, and a product whose unit is `liter` may accept `ml` —
because `PKG_UNITS` already carries those factors and `handleLinkStore` can multiply before sending. Do not
offer an entry unit whose canonical form differs from the product's unit; that is a category error (grams for
a product measured in `st`), and the API's normalizer would reject it anyway.

Keep `PKG_UNITS` itself EXACTLY as it is — same keys, same canonical values, same factors.
`tests/test_static_gates.py` asserts this table against the Python twin in `src/domain/pricing.py`, and
changing one side alone fails the build. That gate is deliberate: a gram-versus-kilogram slip would make one
link look 1000x cheaper than every other and win every comparison.

`handleLinkStore`'s POST body to `/products/{id}/stores` gains `package_size` (the label) and
`package_quantity` (the amount, multiplied by the entry unit's factor when an entry helper was used). Leaving
both empty must remain valid — a link is saveable before its pack size is known (D-02), and the first scrape
autofills it.

  <action>
Add a product links panel to `src/api/templates/admin.html` — a new `modal-product-links` modal built on the
same structure as `modal-price-history`, opened by a new "Links" action button in `buildProductsTable`'s row
actions. Register the button through the existing delegated `data-action` listener; do not use an inline
`onclick` with an interpolated id.

`showProductLinks(productId, productName, productUnit)` fetches `GET /products/{id}/links` and renders a
table, one row per link (D-12, AC#3):

| Butik | Förpackning | Pris | kr/{unit} | Butiken anger | Senast kollad | Actions |

- **kr/{unit}** is the COMPUTED unit price and the ONLY sortable price column. Label it with the product's
  unit ("kr/st", "kr/liter", "kr/kg"). The rows arrive already ranked cheapest-per-unit from the API, so the
  default view is correct with no client-side work.
- **Butiken anger** is `store_unit_price_sek`, the store's PRINTED comparison price, shown as a raw signal and
  rendered visually secondary (muted). It must NOT be sortable and must NOT be the default sort. Store
  definitions differ — one store prints kr/rulle and another kr/100g for the same goods — which is the entire
  reason the computed column exists. Making it clickable would invite exactly the wrong comparison.
- **The "needs amount" flag (D-02):** when `needs_amount` is true, render the kr/unit cell as a call to action
  ("saknar mängd" with the edit affordance) rather than an empty cell. These rows already sort last.
- **The mismatch flag (D-09):** when `quantity_mismatch` is true, badge the Förpackning cell and show both
  numbers — the stored `package_quantity` and the page's `scraped_package_quantity` — so the operator can see
  what the page claims without the app having silently adopted it. The flag is derived and self-clears the
  moment either number is corrected; do not add a dismiss control.
- **Empty state:** a product with zero links is legitimate (the product is the abstract good; links accrue).
  Render an empty state pointing at the Link button, not an error.

Add a `sortLinksBy(column)` handler for client-side column sorting over the already-fetched rows (kr/unit,
price, store). With at most a handful of links per product this is the smallest correct implementation of
"sortable", and the server-side ranking remains the source of truth for the default order. Rows with a null
kr/unit must always sort LAST regardless of direction — a link with no amount is not the cheapest one.

Row actions, wired to the endpoints 04.1-04 created:
- **Edit packaging** — opens a small form and sends `PUT /product-stores/{id}/packaging` with `package_size`
  and `package_quantity`, reusing the same auto-label chain as the link dialog. This is how a mismatch gets
  resolved and how a "needs amount" link gets its number.
- **Check now** — `POST /check/{id}` (the endpoint already exists). On success, re-render the panel: the
  response carries the possibly-autofilled quantity and any mismatch message, so the operator sees the scrape
  fill the amount in place. This is AC#4 made visible.
- **Unlink** — `DELETE /product-stores/{id}` (re-keyed in 04.1-04), with a confirm.

Run every interpolated value through `escapeHtml`.
