# Phase 04.1 Plan 02: Scraper Reports Evidence, Not Answers — Summary

The scrape stopped being the source of the unit-price ANSWER and became the source of
package-quantity EVIDENCE: `store_unit_price_sek` now provably contains only what the store
printed, and `package_amount` + `package_unit` carry the quantity signal that makes non-countable
goods (a 0.5-litre bottle, a 400 g bag) verifiable — not just multipacks.

## The final `PriceExtractionResult` field list

04.1-03 constructs this in every scrape write-path. There are **no defaults** — every construction
must pass all ten fields:

```python
@dataclass
class PriceExtractionResult:
    price_sek: Decimal | None
    store_unit_price_sek: Decimal | None  # RENAMED. The store's PRINTED comparison price (D-05)
    offer_price_sek: Decimal | None
    offer_type: str | None
    offer_details: str | None
    in_stock: bool
    confidence: float
    pack_size: int | None                 # kept: "16-p" style item count
    package_amount: Decimal | None        # NEW (D-08): 0.5, 400, 24
    package_unit: str | None              # NEW (D-08): "st" | "ml" | "l" | "g" | "kg", lowercased
    raw_response: dict[str, str | float | bool | None]
```

It satisfies `pricing.ScrapedPackage` structurally (`package_amount`, `package_unit`, `pack_size`),
so it can be handed straight to `apply_scrape_to_link(link, extraction, product_unit)`. Verified:
a `400 g` reading against a `kg` product yields `scraped_quantity_from(...) == Decimal("0.40")`.

## What Was Built

### `src/domain/parser.py` — three coordinated changes

1. **The `unit_price` prompt bullet was inverted.** It used to *instruct the model to compute*
   ("If the page shows an irrelevant unit price (like kr/kg for toilet paper), calculate it
   yourself: unit_price = price / pack_size"). It now asks for the jämförpris **exactly as
   printed**, explicitly including the case where its unit differs from the pack unit, says "Do NOT
   compute it", and returns null when the page prints none. Two new bullets request
   `package_amount` and `package_unit`.
2. **The `price / pack_size` synthesis was deleted from the decoder.** This is the whole point of
   the plan. The prompt bullet and the decode fallback were the *same contract* stated twice —
   changing one without the other would have quietly reintroduced synthesized values.
3. **The low-confidence discard branch** (`parser.py:178`) also constructs a result; it was updated
   to the new field names with `None` for both package fields. Missing this is how a rename ships
   half-done.

### Extractors — rename only, semantics deliberately untouched

`willys_api.py`'s `comparePrice` ("33,29 kr/kg") was *already* exactly D-05's semantics — the
store's printed compare price. Only the field name changed. It is **not** normalized, **not**
computed, and no quantity is back-derived from it (out of scope; D-07's never-overwrite rule
governs that either way). `jsonld.py` is `None` as before. `base.py` only type-annotates
`PriceExtractionResult` and names no field — correctly left alone.

### The regression guard

`test_no_printed_unit_price_is_not_synthesized` feeds the parser a response with a price (139.90)
and a pack size (24) but **no** printed unit price, and asserts `store_unit_price_sek is None`. If
anyone reinstates the fallback, that test fails on the value `5.829…`. This is what stops the
"Store says" column from silently becoming a mix of store claims and our own arithmetic — the one
signal it exists to provide (spotting the Coop kr/100g mismatch).

## Deviations from Plan

### 1. [Rule 1 - Blocking] `ruff` does not exist in this project — substituted `python -m compileall`

- **Found during:** Task 1 (and pre-empted by the orchestrator's ruling and 04.1-01's Deviation 1).
- **Issue:** Both `<verify>` blocks end in `poetry run ruff check`. Ruff is not in
  `pyproject.toml`, not in `poetry.lock`, not on PATH, and has no config in the repo.
- **Resolution:** Substituted `poetry run python -m compileall -q <files>` (clean on all six
  files). **Did not install it** — package installs are excluded from auto-fix, and adding a linter
  mid-phase would mutate `poetry.lock` and surface pre-existing findings in files this plan does
  not own.

### 2. [Rule 1 - Plan bug] The "domain clean" acceptance assert is unsatisfiable as written

- **Found during:** Task 2 acceptance.
- **Issue:** The criterion asserts the only surviving bare `unit_price_sek` under `src/domain/` is
  `notifier.py`. In fact four files match: `notifier.py` (deliberate — the render kwarg),
  `scheduler.py` and `service.py` (**owned by 04.1-03** — the criterion's own prose says so, so the
  assert describes the state *after the next plan*, not this one), and `models.py`, which matches
  only because the regex lookbehind excludes `store_` but not `computed_` — every hit there is
  `computed_unit_price_sek`, the 04.1-01 hybrid that was *deliberately* named to avoid reusing the
  dropped name. A **false positive**, not a relic.
- **Resolution:** Did not weaken anything and did not reach into 04.1-03's files to force the
  assert green. Verified the satisfiable invariant the criterion was reaching for:
  - the five files this plan owns contain **zero** bare `unit_price_sek` (asserted, passes);
  - **every** `PriceExtractionResult` construction in `src/` uses `store_unit_price_sek=`
    (all four: parser ×2, willys_api, jsonld).
- **Note for 04.1-03:** after it lands, the residual set should be exactly `notifier.py` +
  `models.py` (the latter being the `computed_` false positive). If a future gate reuses this
  regex, tighten the lookbehind to `(?<!store_)(?<!computed_)`.

No other deviations. The field names, prompt wording, decode idiom, and the Willys expected values
(33.29 / 12.50 / None — unchanged, not weakened) are exactly as the plan specified.
