# Phase 04.1 Plan 04: The Link Gets Its Own Address â€” Summary

The three endpoints that resolved a link by `(product_id, store_id)` are gone. Links are now
addressed by their own id, a link's packaging is editable, a duplicate URL is a 409 instead of a
500 carrying a driver message, and a static gate fails the build if the pair-keyed shape ever
comes back.

## The routes 04.1-07 wires the UI to

`product_store_id` is already present in every row dict the product page renders
(`admin.py` store_data `"product_store_id"`), so the UI needs no extra fetch to call these.

| Method | Path | Body | Returns |
|---|---|---|---|
| `PUT` | `/product-stores/{product_store_id}/frequency` | `{"check_frequency_hours": int (72-240, required), "check_weekday": int\|null (0-6)}` | `{"message", "next_check_at"}` |
| `PUT` | `/product-stores/{product_store_id}/packaging` | `{"package_size": str\|null, "package_quantity": float\|null}` | `{"message", "package_size", "package_quantity"}` |
| `DELETE` | `/product-stores/{product_store_id}` | â€” | `{"message"}` |
| `POST` | `/products/{product_id}/stores` | **gains** `package_size`, `package_quantity` | `{"product_store_id", "message"}` |
| `POST` | `/products` | **loses** `package_size`, `package_quantity`; keeps `unit` | `{"product_id", "message"}` |

All three link routes: malformed uuid â†’ **400**, unknown link â†’ **404**. Link create: a
`package_quantity` â‰¤ 0 â†’ **400**; a `store_url` that is already tracked â†’ **409**.

**REMOVED (do not call):** `PUT /products/{pid}/stores/{sid}/frequency` and
`DELETE /products/{pid}/stores/{sid}`. Deleted outright, not aliased â€” their path *is* the
ambiguous key, so no alias could have been correct.

## The bug this plan owned

Three code paths resolved a link with
`select(ProductStore).where(ProductStore.product_id == ..., ProductStore.store_id == ...)` â†’
`.scalar_one_or_none()`. That was single-valued **only** because of the `uq_product_store`
constraint 04.1-01 dropped. Add "Lambi 8-pack at Willys" beside the 24-pack â€” *the phase's own
acceptance scenario* â€” and each raises `MultipleResultsFound` â†’ HTTP 500.

| Site | Was | Now |
|---|---|---|
| `PUT .../frequency` (admin.py:596) | pair-keyed | `ProductStore.id == ps_uuid` |
| `DELETE .../stores/{sid}` (admin.py:683) | pair-keyed | `ProductStore.id == ps_uuid` |
| bulk-import link dedupe (admin.py:1539) | pair-keyed | `ProductStore.store_url == store_url` (the new natural key) |

The frequency endpoint's body contract, its validation ranges (72â€“240 hours, weekday 0â€“6), and
its `next_check_at` recomputation (both the weekday branch and the jitter branch) are byte-for-byte
what they were. **Only the addressing changed.**

## The gate can actually fail â€” and it was proven against the real file

`test_no_link_lookup_by_product_store_pair` is an **AST** detector, not a regex. It flags a
statement that both `select(ProductStore)` and constrains `ProductStore.product_id` **and**
`ProductStore.store_id` against non-PK operands.

The distinction matters: `ProductStore.store_id == Store.id` and
`ProductStore.product_id == Product.id` are **join conditions** and appear all over `admin.py`
and `service.py` (including in `POST /check/{product_store_id}`, the correct pattern). A gate
that flagged either identifier alone would be permanently red and switched off within a week.
The AST sees the difference structurally; a regex cannot. It also ignores comments and strings
by construction â€” 04.1-02 and 04.1-03 each lost time to a source-text gate tripping on prose.

Beyond the plan's required `_BAD_SAMPLE` self-check (plus a `_GOOD_SAMPLE` must-not-flag
counterpart), I **mutation-verified the gate against the real pre-phase `admin.py`**
(`git show HEAD~2:src/api/admin.py`). It reports exactly three findings, at exactly the three
lines the plan named:

```
['PRE-PHASE admin.py:596', 'PRE-PHASE admin.py:683', 'PRE-PHASE admin.py:1539']
```

That is the difference between a gate that is *asserted* to work and one that is *shown* to.

**Known blind spot (inherited, documented):** the detector sees SQL-level shapes. `get_current_deals`'s
old Python-side `(product.id, store.id)` tuple dedupe â€” the fifth pair-keyed site, fixed in
04.1-03 â€” is structurally invisible to it. 04.1-03's `test_deals_two_links_same_store` is that
site's guard.

## What Was Built

### `src/api/schemas.py`

- `ProductCreate` / `ProductUpdate` / `ProductResponse` â€” package fields **removed**. `unit` stays
  on `ProductCreate` (the canonical comparison unit is a property of the good, not the package).
  `ProductCreate`'s docstring used to *instruct the reader to create a separate product per size
  variant* â€” the abolished model, stated inside the API contract itself. Rewritten.
- `ProductStoreLink` â€” gains `package_size` + `package_quantity`.
- **`ProductStoreUpdate` (new)** â€” the packaging-edit body. Exposes **neither `store_id` nor
  `store_url`** (T-04.1-14): the URL is the link's identity now, and silently re-pointing a link
  at a different page would rewrite the meaning of its entire price history. A test asserts a
  `store_url` in the body is ignored, not applied.
- `PricePointResponse` â€” keeps `unit_price_sek` (its **meaning** changed: it is the COMPUTED value,
  the only one that may be sorted on) and gains `store_unit_price_sek` (what the store printed â€”
  display only). Both are commented as such; they are two different numbers.

### `src/api/admin.py`

- The three re-keyed sites above; `POST /check/{product_store_id}` copied verbatim as the pattern.
- **`PUT /product-stores/{id}/packaging`** â€” D-11 needs a way to edit a link's amount and label and
  none existed.
- `create_product` â€” the `package_quantity > 0` block, the `Decimal` conversion, and both kwargs
  removed. **The validation did not vanish â€” it MOVED** to both link write paths (see below).
- `update_product` â€” package branches deleted.
- `link_product_to_store` â€” validates `package_quantity > 0` (400), passes both fields through,
  and catches `IntegrityError` â†’ **409** in an arm placed *above* the blanket `except Exception`.
- Bulk import â€” product branches no longer set package attributes on `Product`; the link branch
  sets them on `ProductStore` from the link entry (`package_size`, `package_quantity` are now
  valid keys in an import link entry).

## Deviations from Plan

### 1. [Rule 3 - Blocking] `ruff` does not exist in this project â€” substituted `python -m compileall`

- **Found during:** pre-empted by the orchestrator's ruling; 04.1-01/-02/-03 all hit it.
- **Issue:** every `<verify>` block ends in `poetry run ruff check`. Ruff is not in
  `pyproject.toml`, not in `poetry.lock`, not on PATH.
- **Resolution:** `poetry run python -m compileall -q <files>` (clean on all four). **Did not
  install it** â€” package installs are excluded from auto-fix; `poetry.lock` untouched.

### 2. [Rule 2 - Missing critical functionality] The static gate is AST-based, not regex-based

- **Found during:** Task 3.
- **Issue:** the plan describes a "source scan with comments stripped". A text scan cannot
  distinguish `ProductStore.store_id == Store.id` (a join â€” legitimate, ubiquitous) from
  `ProductStore.store_id == store_uuid` (the bug). The plan itself flags this hazard and warns
  the naive gate "would be permanently red and get disabled".
- **Fix:** implemented the detector over `ast`, matching the *shape* the plan specifies
  (a `select(ProductStore)` statement constraining **both** columns as filters). Comments and
  strings are excluded structurally rather than by pre-stripping.
- **Files:** `tests/test_static_gates.py`. Verified against the real pre-phase file, not just
  the sample.

No other deviations. Both old routes are gone from the route table; every acceptance criterion
in all three tasks was executed as written (with the ruff substitution) and passed.
