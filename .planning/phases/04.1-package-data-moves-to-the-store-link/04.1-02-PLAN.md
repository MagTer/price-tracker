<objective>
Invert the scraper's role: it stops being the source of the unit-price ANSWER and becomes the source of
package-quantity EVIDENCE (D-06).

Purpose: MODEL-04 says `store_unit_price_sek` is the store's PRINTED value. Today `parser.py` computes a
fallback (`price / pack_size`) whenever the page prints nothing — which would turn the "Store says" column
into a mix of store claims and our own arithmetic, destroying the one signal it exists to provide (spotting
the Coop kr/100g mismatch). MODEL-05 needs `package_amount` + `package_unit` so non-countable goods
(a 0.5-litre bottle, a 400 g bag) can be verified too, not just multipacks.

Output: reshaped `PriceExtractionResult`; a parser prompt and decoder that report rather than compute; both
extractors renamed onto the new field; updated parser and extractor tests.
</objective>

**`src/domain/parser.py`** — three changes:

1. `_build_prompt` — REWRITE the `"unit_price"` bullet. It currently instructs the model to COMPUTE
   ("If the page shows an irrelevant unit price (like kr/kg for toilet paper), calculate it yourself:
   unit_price = price / pack_size"). That instruction now directly contradicts D-05. Replace it with an
   instruction to report the comparison price (jämförpris) EXACTLY as printed on the page, including when

   null if not stated). Keep the existing `"pack_size"` bullet unchanged.

2. `_extract_with_model` decoder — DELETE the fallback that computes a unit price from `price / pack_size`
   when the model returned none. Let the value stay `None`. That branch is what would poison the "Store
   says" column with our own arithmetic; the computed column (from `pricing.unit_price_py`) already covers
   the case where the page prints nothing. Decode the two new fields with the same guarded idiom the other
   fields use (`Decimal(str(...))` / `str(...)` only when the key is present and truthy, else `None`);

## Note: notifier.py is deliberately NOT modified

The seed and CONTEXT.md both pointed at `notifier.py` for the unit-price work. That was wrong (RESEARCH.md
Pitfall 2): `notifier.py` only RENDERS the alert email. Its `unit_price_sek` keyword argument is a display
parameter and keeps its name; the VALUE passed to it becomes the computed unit price, and that change lands
in `scheduler.py` in 04.1-03. Do not touch `notifier.py` in this plan.
