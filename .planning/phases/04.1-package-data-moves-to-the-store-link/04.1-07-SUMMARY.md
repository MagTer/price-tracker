# Phase 04.1 Plan 07: The Screen That Answers The Toilet-Paper Question — Summary

A product's links are now on one screen with package label, price, and a **computed, sortable
kr/unit** — with the store's printed unit price beside it, muted and deliberately un-sortable.
The packaging chain moved from the product dialog to the link dialog, where the package
actually lives. **The Python suite is still 202 passed / 0 failed; no Python was changed.**

## The gate that would have shipped vacuous

`src/api/templates/admin.html` has **no `<script>` tag**. It is a three-part fragment split on
`<!-- SECTION_SEPARATOR -->` and reassembled at `admin.py:1659`. The plan's original gate
extracted the JS by grepping for a script tag, got **zero bytes**, and `node --check` **exits 0
on empty input** — so it would have passed no matter what JavaScript was written, on the largest
net-new surface in the phase.

The corrected extraction (in the plan as executed) pulls the real thing:

```
awk '/SECTION_SEPARATOR/{n++;next} n==2' src/api/templates/admin.html > /tmp/admin.js
test -s /tmp/admin.js && node --check /tmp/admin.js
   -> 20,660 bytes, js syntax ok
```

The `test -s` guard means a future empty extraction **fails** instead of silently passing.

## The sort assertion is the load-bearing one — and it was executed, not eyeballed

`sortRows(rows, column, direction)` is a pure, top-level function: takes an array, returns a new
one, touches no DOM and no globals. That is not stylistic — the rest of the file calls
`document.getElementById()` at top level and **cannot be evaluated in node at all**, so the
testable logic had to be separable or it could not be tested.

Driven over `[7.49, null, 5.83]` in **both** directions:

```
asc :  [5.83, 7.49, null]
desc:  [7.49, 5.83, null]
purity: input untouched -> [7.49, null, 5.83]
```

**Mutation-verified.** I did not trust the plan's claim that the harness catches the naive
comparator — I ran it:

```
function sortRows(rows, column, direction) {           // the naive version
  const dir = direction === 'desc' ? -1 : 1;
  return rows.slice().sort((a,b) => ((a[column]||0) - (b[column]||0)) * dir);
}
-> MUTATION CAUGHT in asc -> order was [null, 5.83, 7.49]
```

That is the bug in one line: **a link with no amount sorts FIRST — an un-priced link
masquerading as the cheapest thing on the page.** The screen would confidently recommend it.
It is exactly the failure the "needs amount" flag (D-02) exists to prevent, and it looks
completely fine on inspection, which is why the assertion has to be *run*.

## What the panel shows

| Butik | Förpackning | Pris | kr/{unit} | Butiken anger | Senast kollad | Actions |

- **kr/{unit}** — the COMPUTED unit price (D-03), labelled with the product's own unit
  ("kr/st", "kr/liter", "kr/kg"), and the only sortable price column. Rows arrive
  **already ranked cheapest-per-unit with nulls last from the service**, so the default view
  re-sorts nothing — the domain owns the one unit-price definition, and a second sort here is
  how a second definition gets in.
- **Butiken anger** — `store_unit_price_sek`, what the store PRINTED. Muted, and it carries
  **no `data-sort` attribute**, so it structurally cannot be sorted on. One store prints
  kr/rulle and another kr/100g for the same goods (D-05's Coop example); making this column
  clickable would invite precisely the wrong comparison and the screen would recommend the
  wrong pack with total confidence.
- **needs_amount** → renders `saknar mängd` as a button that opens the packaging editor.
  **Never a blank cell** — a blank reads as "free" and sorts like zero.
- **quantity_mismatch** → badges the Förpackning cell with **both** numbers ("sidan säger 12 —
  du har 24"). The app has not adopted the page's reading: the typed value is *intent*, the page
  is only *evidence* (D-07). The flag is derived and self-clears the moment either number is
  corrected, so there is no dismiss control.
- **Empty state** — a product with zero links is legitimate (the product is the abstract good;
  links accrue), so it points at the Link button rather than reading as an error.

Row actions address links **by the link's own id**, via the endpoints 04.1-04 re-keyed:
`PUT /product-stores/{id}/packaging`, `POST /check/{id}`, `DELETE /product-stores/{id}`.
"Check now" re-fetches the panel afterwards, so the operator **watches the scrape fill the
amount in place** — AC#4 made visible.

## The dialogs

**Product (D-10, AC#1):** name / brand / category / **unit only**. Unit is a plain dropdown of
the three canonical units (st / liter / kg) — `Product.unit` now stores exactly one of those and
nothing else, so the ml/g entry helpers do not belong here any more: they exist to normalize an
AMOUNT, and the product no longer has one. `handleCreateProduct`'s POST body sends neither
`package_size` nor `package_quantity`.

**Link (D-11, AC#2):** gains the chain — amount + auto-suggested editable label — and has **no
unit selector** (verified: the form's only `<select>` is the store picker). The product already
declares its unit, so the amount field is *labelled in it* ("Amount in package (st)"), the step
is integer for `st` and 0.01 otherwise, and the label suggestion comes from the matching
`PKG_UNITS` entry. Entry helpers survive as **buttons** (not a select) offering only units whose
canonical form IS the product's — `l`/`ml` for a liter product, `kg`/`g` for a kg product,
nothing for `st`. Grams are never offered for a product counted in `st`; that is a category error
and the API's normalizer would reject it anyway.

Both package fields stay optional — no `required` attribute — so a link is still saveable before
its pack size is known (D-02), and the first scrape autofills it.

`PKG_UNITS` is **byte-identical**: same keys, same canonical values, same factors.
`tests/test_static_gates.py` asserts it against the Python twin in `domain/pricing.py` and still
passes. That gate is deliberate: a gram-versus-kilogram slip makes one link look 1000× cheaper
than every other and win every comparison it enters.

## Deviations from Plan

### 1. [Rule 2 — Missing critical functionality] The chain is prefix-keyed, not copied

- **Found during:** Task 2.
- **Issue:** the plan says the packaging chain "moves" to the link dialog and reuses
  `pkgLabelDirty` "verbatim". But Task 2 then requires a **second** dialog running the same
  chain (Edit packaging, which "reus[es] the same auto-label chain as the link dialog"). A
  verbatim copy means two copies of the g/kg factor arithmetic.
- **Fix:** the chain functions take a dialog prefix (`pkgFieldsChanged('link')` /
  `pkgFieldsChanged('edit')`), and the `pkgLabelDirty` latch became per-dialog state in
  `pkgChain(prefix).labelDirty`. Behaviour is identical; there is one implementation. Written
  this way from Task 1 so Task 2 added no chain code at all.
- **Rationale:** a second copy of the factor table is exactly how the two drift apart, and drift
  in *that* table is the 1000×-cheaper bug the static gate exists to catch.

### 2. [Discretion, sanctioned by the plan] Entry helpers are buttons, not a select

AC#2 forbids a unit `<select>` on the link dialog, but D-11 keeps the entry helpers. Rendering
them as small toggle buttons satisfies both: no selector, still able to type 500 g for a product
measured in kg. The plan explicitly leaves entry-helper presentation to discretion.

### 3. [Observation, not a deviation] `apiPut` did not exist

The file had `apiGet` / `apiPost` / `apiDelete` but no `apiPut` — nothing in the UI had ever
called a PUT endpoint (consistent with 04.1-04's finding that the re-keyed routes had **no JS
caller at all**). Added it in the house shape.

No other deviations. Every acceptance criterion in both tasks was executed as written and passed.
