# Phase 04.1 Plan 01: Model Foundation Summary

One canonical definition of unit price (`price / package_quantity`, Python and SQL sides sharing
one implementation), the package columns moved from `Product` to `ProductStore`, `store_url` made
the link's natural key, and `0001_initial.py` rewritten in place as the DDL image of the result.

## What Was Built

### `src/domain/pricing.py` (new) — exact signatures for downstream plans

```python
PKG_UNITS: dict[str, tuple[str, Decimal]]   # entry unit -> (canonical unit, factor)
#   st -> ("st", 1) | ml -> ("liter", 0.001) | l -> ("liter", 1) | g -> ("kg", 0.001) | kg -> ("kg", 1)
CANONICAL_UNITS: tuple[str, ...]            # ("st", "liter", "kg")
MAX_PACKAGE_QUANTITY: Decimal               # Decimal("99999999.99") — the Numeric(10,2) ceiling

class ScrapedPackage(Protocol):             # PriceExtractionResult satisfies this after 04.1-02
    package_amount: Decimal | None
    package_unit: str | None
    pack_size: int | None

class LinkLike(Protocol):                   # ProductStore satisfies this
    package_quantity: Decimal | None
    scraped_package_quantity: Decimal | None

def unit_price_py(price: Decimal | None, qty: Decimal | None) -> Decimal | None
def unit_price_expr(price_col: ColumnElement[Any], qty_col: ColumnElement[Any]) -> ColumnElement[Any]
def normalize_amount(amount: Decimal | None, entry_unit: str | None, product_unit: str | None) -> Decimal | None
def scraped_quantity_from(extraction: ScrapedPackage, product_unit: str | None) -> Decimal | None
def apply_scrape_to_link(link: LinkLike, extraction: ScrapedPackage, product_unit: str | None) -> str | None
def quantity_mismatch(link: LinkLike) -> bool
```

`apply_scrape_to_link` returns a mismatch message (page value first, stored value second) or
`None`, and **never** overwrites a non-NULL `package_quantity`. It always assigns
`link.scraped_package_quantity`, including when the reading is `None`. The module imports nothing
from the ORM — models imports *it*, and the reverse would be a cycle.

### `src/domain/models.py` (reshaped)

- **`Product`** — `package_size` / `package_quantity` deleted; `unit` stays. Docstring rewritten:
  it previously taught "a specific SKU including package size / different package sizes are
  separate products", which is the exact model this phase abolishes.
- **`ProductStore`** — gains `package_size` (`String(50)`), `package_quantity` (`Numeric(10,2)`,
  nullable per D-02) and `scraped_package_quantity` (`Numeric(10,2)`, nullable per D-09).
  `__table_args__` swapped: `uq_product_store(product_id, store_id)` →
  `UniqueConstraint("store_url", name="uq_product_stores_store_url")`.
- **`PricePoint`** — `unit_price_sek` deleted; `store_unit_price_sek` added (D-05, never sorted on).
  Two hybrids added: `effective_price_sek` (offer wins, `coalesce` on the SQL side) and
  `computed_unit_price_sek` (correlated scalar subquery on the SQL side — works with no join).
  Both `MissingGreenlet` and redundant-subquery hazards are recorded as comments at the hybrid.

Verified against RESEARCH.md's table: 24-pack at 139.90 → `5.829166…` unrounded; offer 39.90 →
`1.6625`; NULL quantity → `None`, no crash.

### `alembic/versions/0001_initial.py` (rewritten in place, D-14)

Revision id unchanged (`0001_initial`, `down_revision` None) — no 0002 stacked, one migration
module on disk. `products` loses both package columns; `product_stores` gains all three and the

## Deviations from Plan

### 1. [Rule 3 - Blocking, NOT auto-fixed] `ruff` is not installed and is not a project dependency

- **Found during:** Task 1, running the plan's first `<verify>` block.
- **Issue:** Every acceptance block in this plan (and in 04.1-02, -03, -08) ends with
  `poetry run ruff check …`. Ruff is **not in `pyproject.toml`**, not in `poetry.lock`, not on
  `PATH`, and has no config file anywhere in the repo. `poetry run ruff --version` →
  `Command not found: ruff`. The planner assumed a lint gate this project has never had.
- **Resolution:** **Did not install it.** Package-manager installs are explicitly excluded from
  auto-fix. Adding ruff would also mutate `poetry.lock` mid-phase and likely surface pre-existing
  lint findings across files this plan does not own — out of scope either way.
- **Substituted:** `poetry run python -m compileall -q <files>` (clean) plus the full pytest runs.
  Every other acceptance criterion in all three tasks was executed and passed as written.
- **Action needed:** the operator should decide whether to add ruff as a dev dependency. Plans
  **04.1-02, 04.1-03 and 04.1-08 carry the same broken assumption** and will hit this too.

### 2. [Rule 2 - Missing critical functionality] `normalize_amount` also rejects quantities that round to zero

- **Found during:** Task 1.
- **Issue:** The plan specifies rejecting a result that "is not strictly positive or exceeds
  `MAX_PACKAGE_QUANTITY`", then quantizing to two decimals. A 1 g reading against a `kg` product
  is `0.001` — strictly positive, in range, but it quantizes to `0.00`, which would then be
  stored as a **zero quantity**. That is precisely the divide-by-zero input `NULLIF` exists to
  neutralize, and it would render the link permanently as "needs amount" with a bogus 0 on file.
- **Fix:** re-check `> 0` *after* quantizing; return `None` if the value rounds away.
- **Files:** `src/domain/pricing.py` (`normalize_amount`), covered by the range tests.

No other deviations. The column specs, constraint name, revision id, and every export signature
are exactly as the plan specified.
