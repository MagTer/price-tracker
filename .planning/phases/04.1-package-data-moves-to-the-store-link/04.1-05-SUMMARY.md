# Phase 04.1 Plan 05: MCP `compare_stores` Goes Per-Link — Summary

`compare_stores` now returns one row per LINK, ranked by the computed kr/unit, with the store's
printed comparison price in its own never-sorted column — and its Jämförspris column shows a real
number for the first time since the extraction.

## ⚠ FLAG FOR THE OPERATOR — this plan fixes a silently broken column

**`compare_stores` has printed `N/A` in its Jämförspris column and `Nej` in its Lager column for
EVERY row, on every call, since the extraction.** `server.py:147` read `row.get("unit_price_sek")`
and `row.get("in_stock")` off `get_price_history()` rows — and **neither key had ever been in that
dict**. `check_price` read the same dead `in_stock` key, so it has reported "slut i lager" for every
product it has ever been asked about.

The `priser` skill has therefore never once seen a comparison price, and has never once seen a
correct stock status. This was not a missing feature; it was a dead column that read as data. 04.1-03
added the keys; this plan is the first call that actually shows them.

## What the tool renders now

```
| Butik  | Förpackning | Pris      | Erbjudande | Jämförspris (kr/st) | Butiken anger | Lager |
|--------|-------------|-----------|------------|---------------------|---------------|-------|
| Willys | 24-pack     | 139.90 kr | N/A        | 5.83 kr             | 5.83 kr       | Ja    |
| ICA    | 8-pack      | 59.90 kr  | N/A        | 7.49 kr             | 8.10 kr       | Ja    |
| Coop   | —           | 129.00 kr | N/A        | saknar mängd        | 12.90 kr      | Ja    |
```

That is the actual rendered output (executed, not sketched). Read it against D-05: **ICA's computed
7.49 and its printed 8.10 disagree**, and the printed number is the one that would have won a sort.
Coop's link has no amount yet, so it says `saknar mängd` rather than a bare `N/A` — a marker the
agent (and through it the operator) can act on — and it already sorts last because the service put
it there.

### `compare_stores` (D-13)

- Data source switched from `get_price_history()` to **`get_links_for_product()`**: one row per link,
  already ranked cheapest-per-unit with NULLs last, already carrying the computed value. The tool is
  now a **render** — **no sort and no unit-price arithmetic exist in `server.py`** (grep-verified;
  `seen_stores` is gone).
- Columns: Butik | Förpackning | Pris | Erbjudande | **Jämförspris (kr/{product.unit})** | **Butiken
  anger** | Lager. The computed value and the store's printed value are structurally separate columns
  and the printed one is never a sort key (D-05, T-04.1-15).
- Docstring rewritten. It is the agent's only description of the tool, so it now states explicitly
  that two pack sizes at one store are two rows, that Jämförspris is the computed and only comparable
  number, and that "Butiken anger" must never be used to rank.
- Empty-link case renders a sentence, not a headerless table.

### `check_price` (incidental fix)

- Dedupes on `product_store_id`, not store name — two pack sizes at one Willys no longer collapse
  into an arbitrary winner — and labels each row `Willys (24-pack)`.
- Reports real stock. Its store-level shape is deliberately kept: "what does it cost where" is a
  different question from "which pack is cheapest per unit", and `compare_stores` now answers the
  latter properly.

## Tests

`tests/test_mcp.py` went from 11 tests to 16 (all green; full suite **190 passed, 0 failed**).

**Updated on the merits, not in response to a red bar.** As 04.1-03 warned, this suite mocks the
service layer and was green throughout the model change *while asserting nothing about the new row
shapes* — its old fixtures literally carried `unit_price_sek: None`, which is what a dead key looks
like when nobody notices. The new `_link` / `_history_row` builders mirror the service's key sets from
the 04.1-03 SUMMARY exactly, so the tests can no longer pass against a shape the service never emits.

| Test | Guards |
|---|---|
| `test_compare_stores_one_row_per_link_ranked_by_unit_price` | three links → three rows; 24-pack ranked above 8-pack (catches anyone re-sorting in the tool); computed and printed values asserted **by column index**, not substring |
| `test_compare_stores_marks_link_without_amount` | NULL quantity → `saknar mängd`, sorted last, no crash (MODEL-07, T-04.1-16) |
| `test_compare_stores_two_links_same_store` | the phase's acceptance scenario at the MCP boundary — both Willys rows appear; the old store-name dedupe dropped one |
| `test_check_price_reports_real_stock` | asserts `i lager` — the regression guard for the key that never existed |
| `test_check_price_keeps_two_links_at_one_store_apart` | both packs visible, each labelled |

The column-index assertion is the load-bearing one: a substring check for `"5.83 kr"` would pass even
if the computed and printed values were merged back into a single column, which is exactly the
misinformation T-04.1-15 exists to prevent.

## Deviations from Plan

### 1. [Rule 3 - Blocking] `ruff` does not exist in this project — substituted `python -m compileall`

- **Found during:** pre-empted by the orchestrator's ruling; plans 01–04 all hit it.
- **Issue:** both `<verify>` blocks and two acceptance criteria end in `poetry run ruff check`. Ruff
  is not in `pyproject.toml`, not in `poetry.lock`, not on PATH.
- **Resolution:** substituted `poetry run python -m compileall -q <file>` (clean on both files). **Did
  not install it** — package installs are excluded from auto-fix; `poetry.lock` untouched.

### 2. [Rule 1 - Test bug, self-inflicted] A blanket `"N/A" not in row` assertion tripped on the offer column

- **Found during:** Task 2. The plan's spirit is "the Jämförspris cell is not N/A". I first wrote it as
  `assert "N/A" not in rows[0]`, which failed — because the **Erbjudande** column legitimately prints
  `N/A` when a link has no active offer (`_fmt_price(None)`, the pre-existing house idiom).
- **Resolution:** replaced with a **column-index** assertion on the specific cell. Strictly stronger:
  it now also proves the value is in the *right* column, which the substring form never did.
- **Note:** `Erbjudande: N/A` for "no offer" is pre-existing behavior of `_fmt_price` across all four
  tools. Left alone — changing it is a cosmetic edit to a shared helper and outside this plan's files.

### 3. [Rule 2 - Missing critical functionality] `Lager` renders `?` when `in_stock` is None

- **Issue:** `get_links_for_product()` outer-joins its price points, so a link with **no price point
  yet** returns `in_stock: None`. The plan's `"Ja" if in_stock else "Nej"` renders that as **`Nej`** —
  asserting "out of stock" about a link nobody has ever checked. That is the same class of confidently
  wrong answer as the dead `in_stock` key this plan exists to fix, and re-introducing it in the fix
  would be an unforced error.
- **Fix:** `"?" if in_stock is None else ("Ja" if in_stock else "Nej")`.

No other deviations. Every acceptance criterion in both tasks was executed as written (with the ruff
substitution) and passed.
