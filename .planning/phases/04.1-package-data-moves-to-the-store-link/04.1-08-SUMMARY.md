# Phase 04.1 Plan 08: The Schema, Proven Against A Real Database — Summary

The phase whose deliverable *is* the schema had never once run that schema against a database.
It has now: `alembic upgrade head` is green on a fresh Postgres, `alembic check` reports no drift,
two links at one store persist, a duplicate `store_url` is rejected, and the kr/unit `ORDER BY`
was **executed by a real query planner** instead of compiled and eyeballed.

**Full suite: 214 passed** (202 mocks + 12 integration). **With the database stopped: 202 passed,
12 skipped, exit 0, 2.7 s** — the fast suite still needs no database.

**No deployment was touched. The D-15 reset was NOT run. The only database created or dropped
anywhere in this plan is the throwaway `price_tracker_test`.** The dev database still holds its
tables and its `alembic_version` row (`0001_initial`), verified after the fact.

A `MagicMock` grows any attribute it is asked for, so it structurally cannot notice a dropped
column; a compiled SQL string cannot notice that `NULLS LAST` was omitted; and no mock has an
opinion about a UNIQUE constraint. `aiosqlite` — declared in `pyproject.toml` since the extraction
— cannot substitute either: the models use `postgresql.UUID` and `JSONB`, which do not compile on
SQLite. That is *why* this repo had no `conftest.py`, and it is why the gap survived four phases.

## The skip guard is the load-bearing part

A DB-backed tier can only coexist with a mocks-only suite if the absence of a database is a
**skip**, not a failure and not a hang. Verified by actually stopping the container, not by reading
the decorator:

```
docker compose stop postgres
poetry run pytest -q   ->  202 passed, 12 skipped, 13 warnings in 2.74s   (exit 0)
docker compose start postgres
poetry run pytest -q   ->  214 passed in 4.85s
```

The probe carries an explicit 3-second timeout, and the skip message names the URL it tried (with
the password masked) and how to start one. `docker compose stop` — never `down -v`; no volume was
touched.

## Mutation verification — two gates, and one of them was vacuous

Both new gates passed on their first run, which is precisely when a gate is most likely to be
proving nothing. So I broke the code and checked they noticed.

| Mutation | Caught? |
|---|---|
| add a column to `models.py` that `0001_initial` does not create | ✅ `alembic check` → *Detected added column 'product_stores.drift_canary'* |
| `nulls_last()` → `nulls_first()` — the un-priced link masquerades as cheapest | ✅ order became `['apotea', 'willys', 'ica']` |
| `NULLIF(qty, 0)` → `COALESCE(NULLIF(qty, 0), 1)` — a missing amount treated as the number 1 | ❌ **passed** — the test was vacuous |

**The third one is the finding.** My first fixture priced the NULL-quantity link at 99.00 kr. With
the missing quantity coalesced to 1, that link's "unit price" becomes 99.00 — which still sorts
*last*, because it is dearer than the 8-pack's 7.49. The bug was fully present and the test was
green. The fixture now prices that link at **4.90**, cheaper than everything else on the page, so
any code that treats a missing amount as a number must rank it **first** — and the test fails.
Re-run: both mutations caught.

One honest negative: **deleting `.nulls_last()` outright is undetectable here**, because Postgres
already sorts NULLs last on an ascending sort. The call is documentation, not behaviour, on this
dialect and in this direction — it earns its keep only if someone flips the sort to `DESC`. The
test proves the *outcome* (the un-priced link is last), which is the thing that matters.

## The near-miss that would have dropped the dev database

`alembic/env.py` prefers the `DATABASE_URL` **environment variable** over `sqlalchemy.url` (the
deployment override added in Phase 3). The plan says to run Alembic programmatically "with the
sqlalchemy URL overridden to the test URL" — which, on any machine where `DATABASE_URL` is
exported, would have migrated **the dev database** while the tests happily talked to the test one.
`alembic_config()` therefore sets both, and restores the environment afterwards. Combined with
forcing the database *name* to `price_tracker_test` (T-04.1-24), the fixture cannot reach the dev
or deployed database even if `DATABASE_URL` points straight at it.

## Deviations from Plan

### 1. [Rule 3 — Blocking] `ruff` does not exist in this project — substituted `python -m compileall`

Carried forward from the orchestrator's ruling; plans 01–07 all hit it. Ruff is not in
`pyproject.toml`, not in `poetry.lock`, not on PATH. **Not installed** — package installs are
excluded from auto-fix, and adding a linter now would surface pre-existing findings across files
this phase does not own. `poetry run python -m compileall -q src tests` → clean. The operator
should decide whether to add ruff as a dev dependency in a follow-up.

### 2. [Rule 1 — Bug in the plan] The mandated `-k duplicate_url` selector matched nothing

The plan names the test `test_duplicate_store_url_rejected` **and** requires
`pytest -k duplicate_url -x` to pass. `duplicate_url` is not a substring of
`duplicate_store_url`, so the selector deselected all 12 tests and the acceptance criterion was
unfalsifiable (`12 deselected` exits 0 — it *looks* like a pass). VALIDATION.md pins the same
`-k duplicate_url` selector for MODEL-02, so the selector is the contract: the tests are named
`test_duplicate_url_rejected` and `test_duplicate_url_rejected_across_products`.

### 3. [Rule 2 — Missing critical functionality] `alembic_config()` must also set `DATABASE_URL`

See "the near-miss" above. Without it, a developer with `DATABASE_URL` exported runs the
migration against their dev database. The plan specified only the `sqlalchemy.url` override.

### 4. [Scope] 12 tests, not the 6 the plan enumerates

The plan's six are all present. Six more were added where the real DB made a claim cheap to check
and impossible to check any other way: the five store seeds survived the 0001 rewrite; a duplicate
URL is rejected even under a *different* product; only a link's LATEST price point ranks it; a link
with no price point at all still appears; a scraped-quantity conflict is flagged without the page's
value being adopted (D-07 in real SQL, not a mock); and the fixture is asserted to be pointed at
`price_tracker_test`.

No other deviations. Every acceptance criterion in all three build tasks was executed as written
(with the ruff substitution) and passed.

## 4. Then, and only then — the D-15 reset. It is yours to run.

The deployed DB is stamped at the old `0001_initial`. Because `0001` was rewritten **in place**, and
Alembic does **not** checksum migration bodies, `alembic upgrade head` will compare equal to head,
**apply nothing, print nothing, and exit 0** — and the app will then run against the **OLD** schema
until the volume is dropped. The first symptom is a runtime error on the first query touching
`product_stores.package_quantity`, long after the deploy looked successful.

**The agent has deliberately NOT run this.** It acts on the live Postgres volume.

The runbook is in `README.md` § "Schema reset (Phase 04.1)", reproduced here verbatim so it
survives even if you never open the README:

```bash
# On the deployment host, in compose/dokploy-apps/price-tracker/ (home-server repo).
# NOT this repo's docker-compose.yml, which is local-dev-only.

# 1. Stop the app and DROP THE POSTGRES VOLUME. This removes the tables AND the
#    alembic_version row that makes the rewritten 0001 look already-applied.
docker compose down -v

# 2. Bring Postgres back up (empty).
docker compose up -d postgres

# 3. Start the app: its command runs `alembic upgrade head`, which now executes the
#    rewritten 0001 against an empty database and re-seeds the five stores.
docker compose up -d

# 4. VERIFY — do not skip; the failure this guards against is silent.
docker compose exec app alembic current   # must print: 0001_initial (head)
docker compose exec app alembic check     # must print: No new upgrade operations detected.
```

`alembic current` printing `0001_initial (head)` **on its own proves nothing** — a stale database
prints the identical line. It is `alembic check` that proves the live schema matches the ORM. If
`check` reports operations, the volume was not actually dropped.

**This destroys all price history.** That is accepted, and it is the basis on which the in-place
rewrite was sanctioned: the database is empty. There is nothing to rescue. *If that is not true —
if there is history on that volume you care about — stop and say so.*
