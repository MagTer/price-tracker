# Phase 04.1 Plan 06: The Admin API Moves onto the Computed Unit Price ‚Äî Summary

Every admin response that shows a unit price now shows the **computed** one, beside ‚Äî never merged
with ‚Äî the one the store printed. "Check now" fills or flags the link's quantity like the scheduler
does. Two pack sizes at one store stopped silently collapsing into one deal. **The full suite is
green again: 202 passed, 0 failed** (was 190) ‚Äî and green for the right reason.

## The green bar was lying, and this is what it was hiding

The suite was **190 passed / 0 failed** when I started. Three separate plans flagged that it was
lying, and it was: `test_api.py` mocked the service layer with `MagicMock`s, and **a MagicMock
auto-creates every attribute it is asked for**. `product.package_size` on a mock returns a mock
instead of raising. So the suite could not see any of this:

| Break | Endpoint | Failure mode |
|---|---|---|
| reads `product.package_size` / `.package_quantity` | `list_products`, `get_product`, `/export` | `AttributeError` at request time |
| reads `latest_price.unit_price_sek` / `pp.unit_price_sek` | `list_products`, `get_product`, `/export` | `AttributeError` ‚Äî column dropped in 04.1-01 |
| reads `price_point.unit_price_sek` **and** omits the now-required `store_unit_price_sek` | `GET /products/{id}/prices` | `AttributeError` **and** `ValidationError` ‚Äî the response could not be constructed at all |
| never called `apply_scrape_to_link` | `POST /check/{id}` | records a price, leaves the quantity blank **forever** |
| deduped on the `(product.id, store.id)` tuple | `GET /deals` | one pack size **silently vanishes** ‚Äî no error, no 500 |

Confirmed against the real model, not by reading the diff:

```
pre-change read would raise: 'Product' object has no attribute 'package_size'
pre-change read would raise: 'PricePoint' object has no attribute 'unit_price_sek'
```

**The fix to the tests is the load-bearing part of this plan.** The new read-path fixtures build
**real (transient, session-less) ORM objects** ‚Äî no DB, the session is still mocked, but
`Product(...)` raises on a dropped column where a mock never would. Every endpoint is then exercised
end-to-end through `TestClient`. These fixtures *can* fail; the old ones could not.

## The row keys 04.1-07 renders

### `GET /products/{product_id}/links` ‚Äî the product page's data source (D-12)

Returns `service.get_links_for_product(product_id)` verbatim: one row per LINK, **ranked
cheapest-per-unit with NULLs last by the SERVICE**. The endpoint renders that order and never
re-sorts ‚Äî the domain owns the one unit-price definition, and a second sort here is how a second
definition gets in. Malformed product id ‚Üí **400**.

```python
{
  "product_store_id": str,        "store_id": str,
  "store_name": str,              "store_slug": str,          "store_url": str,
  "package_size": str | None,     "package_quantity": float | None,
  "scraped_package_quantity": float | None,
  "price_sek": float | None,      "offer_price_sek": float | None,
  "store_unit_price_sek": float | None,   # what the STORE printed (D-05) ‚Äî NEVER sort on this
  "unit_price_sek": float | None,         # the COMPUTED value (D-03) ‚Äî the sortable one
  "in_stock": bool | None,        "checked_at": datetime | None,
  "needs_amount": bool,           # True iff package_quantity is None (D-02)
  "quantity_mismatch": bool,      # derived, self-clearing (D-09) ‚Äî never persisted
}
```

A link with **no price point yet** still appears (the service outer-joins), with `price_sek` /
`unit_price_sek` / `checked_at` all `None` and `needs_amount` still speaking.

### The `stores[]` rows in `GET /products` and `GET /products/{id}`

Same package + price + flag picture, plus the link's scheduling fields
(`check_frequency_hours`, `check_weekday`, `last_checked_at`). These are **not** ranked ‚Äî they come
back in link-creation order. The product page should use `/links`.

## Scrape write-path #3 is closed ‚Äî all three now call the one helper

```
src/domain/scheduler.py: 2   (write-path #1, from 04.1-03)
src/domain/service.py:   2   (write-path #2, from 04.1-03)
src/api/admin.py:        2   (write-path #3 ‚Äî THIS PLAN)
```

`POST /check/{id}` re-implements `service.check_price`'s fetch/parse/record flow **inline** instead
of calling it, so D-07 never reached it for free. It now calls
`apply_scrape_to_link(product_store, extraction_result, product.unit)` before `record_price` ‚Äî
`product` and `product_store` were already in the same row tuple, so this costs nothing. The link is
attached to the session `record_price` commits, so the autofilled quantity and the page's reading
persist with the price point; no second commit.

Exercised for real, not asserted from the diff:

```
POST /check (NULL qty)  -> 200  package_quantity: 24.0, quantity_mismatch: None, unit_price_sek: 5.83
   link.package_quantity  autofilled to: 24.00
   link.scraped_package_quantity        : 24.00

POST /check (conflict)  -> 200  quantity_mismatch: "page says 12.00, you have 24"
   STORED package_quantity still: 24        ‚Üê intent survives the evidence (D-07)
```

Without this, AC#4 ("a scraper run against a real store page fills or verifies the link's quantity")
would have been **false through the UI** ‚Äî the only place the operator ever triggers a check by hand.

## The deals bug that fails silently

`get_current_deals` deduped on a Python-side `(product.id, store.id)` tuple ‚Äî the **fifth**
pair-keyed site, single-valued only because of the constraint D-01 dropped. Fed a 24-pack and an
8-pack at one Willys ‚Äî *the phase's own acceptance scenario* ‚Äî it kept an **arbitrary** one and
discarded the other. No exception. No 500. One pack size just missing from the operator's deals list.

Re-keyed to the link's own id. `DealResponse` gains `package_size` and the computed
`unit_price_sek`; **the recency ordering is kept** (a test asserts the deals are *not* ordered by
unit price ‚Äî exposing the number is not the same as re-ranking the feature).

04.1-04's static gate structurally cannot catch this shape: it detects a `select()` constrained on
both columns, and a Python tuple key is a different shape entirely. So the behavioral test is the
only guard ‚Äî which is why I mutation-verified it.

## Mutation verification ‚Äî both load-bearing tests can actually fail

| Mutation | Result |
|---|---|
| restore `key = (product.id, store.id)` in `get_current_deals` | `test_deals_two_links_same_store` fails: **`assert 1 == 2`** ‚Äî "the two pack sizes at one store collapsed into one deal row" |
| remove the `apply_scrape_to_link(...)` call from `POST /check` | `test_check_autofills_an_empty_link_quantity` fails: **`assert None == Decimal('24.00')`** ‚Äî the quantity is never filled |

Both mutations were reverted; the suite is green. (My first mutation run used bare `python`, which is
not on PATH ‚Äî the "passes" it printed were meaningless. Re-ran under `poetry run python` with an

check that silently no-ops is worse than none, because it manufactures confidence.)

## Deviations from Plan

### 1. [Rule 3 - Blocking] `ruff` does not exist in this project ‚Äî substituted `python -m compileall`

- **Found during:** pre-empted by the orchestrator's ruling; plans 01‚Äì05 all hit it.
- **Issue:** every `<verify>` block and three acceptance criteria end in `poetry run ruff check`.
  Ruff is not in `pyproject.toml`, not in `poetry.lock`, not on PATH.
- **Resolution:** `poetry run python -m compileall -q src tests` (clean). **Did not install it** ‚Äî
  package installs are excluded from auto-fix; `poetry.lock` untouched.

### 2. [Rule 3 - Blocking] Task 1's file-wide gate is only satisfiable after Task 2

- **Issue:** Task 1's acceptance requires `admin.py` to contain **no** `pp.unit_price_sek` /
  `price_point.unit_price_sek` reads *file-wide*. But those reads live in `/export` and
  `trigger_price_check`, which are **Task 2's** declared action ("Do not touch the write paths owned
  by 04.1-04" scopes Task 1 to the three read paths). The gate cannot pass at the Task 1 commit.
- **Resolution:** verified the equivalent gate **scoped to the functions Task 1 owns**
  (`list_products`, `get_product`, `get_price_history`) at commit 1, and ran the **file-wide** gate at
  commit 2, where it passes. No intermediate state is worse than the pre-plan state (which was
  broken in all five paths). A plan-sequencing bug, not a code one.

### 3. [Rule 2 - Missing critical functionality] `bool` was missing from two Pydantic value unions

- **Found during:** Task 1 (`ProductResponse.stores`) and Task 2 (`trigger_price_check`'s return
  annotation, `dict[str, str | float | None]`).
- **Issue:** **`bool` subclasses `int`.** In a smart-mode union of `str | int | float`, Pydantic
  coerces `needs_amount=True` to the integer **`1`**. Observed live: `POST /check` was returning
  `'in_stock': 1.0`, not `true` ‚Äî a **pre-existing** defect on that endpoint, invisible until I
  exercised it. A "needs amount" warning the UI must render as a flag would have arrived as a number.
- **Fix:** added `bool` to `ProductResponse.stores`' value union and to `trigger_price_check`'s
  return annotation. Confirmed `in_stock` now serializes as `true`.
- **Files:** `src/api/schemas.py`, `src/api/admin.py`. Note `schemas.py` is 04.1-04's file ‚Äî the plan
  sanctions touching it for `DealResponse`; this is the same edit class.

### 4. [Observation, not a deviation] `/import` needed no change

The plan's Task 2 says to "align the price-history import rows with the renamed column". **`/import`
never ingested price history at all** ‚Äî verified: `'price_history' in inspect.getsource(import_data)`
is `False`. Export writes it; import drops it. That asymmetry is **pre-existing** and out of this
plan's scope (logged, not fixed). The link package fields were already wired by 04.1-04. So the
import is correct as it stands.

Also noted, not fixed: `/export` now emits `scraped_package_quantity`, and `/import` ignores it. That
is harmless ‚Äî it is *evidence*, regenerated on the next check, and it self-heals.

No other deviations. Every other acceptance criterion in all three tasks was executed as written
(with the ruff substitution) and passed.
