## Link uniqueness rule

Raised by the codebase scout, **not** by the seed: `models.py:96` carries
`UniqueConstraint("product_id", "store_id", name="uq_product_store")` — one link per product
per store. After the move, "Lambi 24-pack at Willys" and "Lambi 8-pack at Willys" are two
links from one product to the same store. The existing constraint forbids exactly the thing
the phase exists to enable. A blocker.

| Option | Description | Selected |
|--------|-------------|----------|
| Unique on `store_url` | A link IS a store page, so the URL is its natural key. Global uniqueness also prevents the same page being tracked twice or attached to two products. | ✓ |
| Unique on (product, store, qty) | Encodes the intent directly, but two different URLs for the same 24-pack collide, and a NULL quantity makes it leaky (NULLs don't compare equal in Postgres). | |
| Drop uniqueness entirely | Simplest migration, but nothing stops a duplicate link quietly doubling the price history. | |

**User's choice:** Unique on `store_url` → D-01

### Should `package_quantity` be NOT NULL on the link?

| Option | Description | Selected |
|--------|-------------|----------|
| Optional, nullable | Add a link fast; let the scraper fill quantity later (AC#4). Links without quantity show no kr/unit. | ✓ |
| Required, NOT NULL | Guarantees kr/unit always computes — but you must know the pack size before you can save a URL, fighting the scraper-autofill criterion. | |

**User's choice:** Nullable → D-02 (later refined, see below)

## Unit-price source of truth

**Key finding surfaced before the question:** the seed says "unit_price_sek = price /
link.package_quantity" as though relocating a calculation. It isn't. `Product.package_quantity`
is never used in any arithmetic today; `PricePoint.unit_price_sek` is 100% LLM-parser output
(`parser.py:261-266`). This phase *changes the source of truth*.

| Option | Description | Selected |
|--------|-------------|----------|
| Computed only | `price / link.package_quantity`, always. One definition → comparable across every store and pack size. Scraped value kept as cross-check signal only. | ✓ |
| Computed, fall back to scraped | Fewer blank cells, but silently mixes two definitions in one sortable column — a "cheapest" row could be an artifact of the store's units. | |
| Keep scraped (today) | Zero change to the pricing path, but leaves the core comparison question unanswered — the phase would ship the model change without the payoff. | |

**User's choice:** Computed only → D-03

**Notes:** The user did not pick blindly — they pushed back with two questions:
*"If we go computed only, that means that part of the scrape is not useful anymore? That also
means that the previous question on nullable package_quantity is impacted?"* Both were correct
and both changed the design:

1. **The scrape is not wasted — its role inverts.** It stops producing the *answer* (unit
   price) and starts producing *evidence about quantity* (`pack_size` from the title;
   quantity back-derived as `price / printed_unit_price`). This is what makes AC#4 real. →
   D-06
2. **Nullable is sharpened, not reversed.** Under computed-only, a link with no quantity has
   *no kr/unit at all* — it is invisible to the core comparison. So nullable cannot mean
   "silently blank forever". → D-02 refined to: nullable in schema, autofilled by first
   scrape, visibly flagged as "needs amount" until then.

### History when a quantity is later corrected (24 → 12)

| Option | Description | Selected |
|--------|-------------|----------|
| Compute on read | Don't persist unit price; derive from `price_sek / link.package_quantity` at display time. Correcting a quantity fixes all history instantly; stale snapshots become impossible. `PricePoint.unit_price_sek` becomes redundant. | ✓ |
| Store snapshot, leave history | Today's shape. Rows recorded before the correction keep the wrong unit price — the kr/unit trend silently contains bad points. | |
| Store snapshot + backfill | Correct history and a fast denormalized column, at the cost of a backfill path that must not be forgotten. | |

**User's choice:** Compute on read → D-04

### Fate of the scraped value / the now-unwritten column

| Option | Description | Selected |
|--------|-------------|----------|
| Repurpose as `store_unit_price_sek` | Drop `unit_price_sek`; add a column holding the store's printed number as-scraped, labeled "what the store claims", shown beside the computed kr/unit and never sorted on. | ✓ |
| Keep only in `raw_data` JSONB | Leanest schema, but the mismatch signal is buried in JSON and nothing surfaces it. | |
| Discard it | Simplest parser contract, but throws away the cross-check that catches a mistyped quantity. | |

**User's choice:** Repurpose as `store_unit_price_sek` → D-05

## Scraper reconciliation on mismatch

AC#4 says "mismatch between page and stored quantity is signal, not noise" — true, but it
never says what the system *does*.

| Option | Description | Selected |
|--------|-------------|----------|
| Autofill empty, flag conflict | NULL → fill silently. Set + disagreement → flag visibly, never overwrite. Typed value is intent; the page is evidence. | ✓ |
| Scraper always wins | Self-healing, but one bad LLM extraction silently rewrites the quantity and (since kr/unit is computed from it) silently corrupts the comparison and the history view. | |
| Autofill empty, ignore conflicts | Simplest — but discards exactly the signal AC#4 asks for; a mistyped 24-for-12 stays wrong forever. | |

**User's choice:** Autofill empty, flag conflict, never overwrite → D-07

### Parser contract — `pack_size: int` can't verify 0.5 liter or 400g

| Option | Description | Selected |
|--------|-------------|----------|
| Extend to amount + unit | Parser also returns `package_amount` (Decimal) + `package_unit` (raw string), normalized with the st/liter/kg factors the admin UI already uses. Makes AC#4 real for liter/kg goods, not just countable ones. | ✓ |
| Only verify `st` products | Smallest change, but liter/kg links never get autofilled or verified — AC#4 half-delivered. | |
| Back-derive from store's unit price | Reuses a value already stored, but only valid when the store's printed definition matches the canonical unit — exactly what can't be relied on (the kr/100g case). | |

**User's choice:** Extend to amount + unit → D-08

### Where the conflict lives

| Option | Description | Selected |
|--------|-------------|----------|
| `scraped_package_quantity` on the link | Every check writes what the page said. UI flags when it differs from `package_quantity`. No state machine; self-clears when either number is fixed. | ✓ |
| `needs_review` boolean + `resolved_at` | A real review queue, but adds state to maintain and can go stale after acknowledgement. | |
| Derive at render from latest PricePoint | Zero schema change, but puts JSON digging in the render path and the signal vanishes if `raw_data` drifts. | |

**User's choice:** `scraped_package_quantity` on the link → D-09

## Migration + rollout shape

| Option | Description | Selected |
|--------|-------------|----------|
| New revision 0002 on top | Honest history and it actually runs on the deployed instance (already stamped 0001). *(Claude's recommendation.)* | |
| Rewrite `0001_initial` in place | Tidiest single migration, as the seed sanctions — but the live DB is stamped 0001, so alembic would see `0001 == head` and apply nothing unless the DB is also reset. | ✓ |

**User's choice:** Rewrite `0001_initial` in place → D-14 — **overriding Claude's
recommendation.** Legitimate given the DB is empty, but it converts the DB reset from an
optional cleanup into a mandatory, named deploy step. Followed up immediately:

### How the deployed instance gets reset

| Option | Description | Selected |
|--------|-------------|----------|
| Drop volume, recreate, `upgrade head` | `docker compose down -v` → recreate → `alembic upgrade head`. Clears tables *and* `alembic_version` in one move. Safe because the DB is empty; it is what makes the in-place rewrite legitimate. | ✓ |
| `alembic downgrade base`, then upgrade | Stays inside alembic, but the downgrade body must still match the OLD 0001 at the moment it runs — which the rewrite is about to destroy. Order-sensitive. | |
| Manual DROPs by hand | Full control, but an undocumented hand operation on a live host; forgetting `alembic_version` is the exact failure that leaves the app on the old schema. | |

**User's choice:** Drop volume, recreate, upgrade head → D-15

### MCP `compare_stores` shape

| Option | Description | Selected |
|--------|-------------|----------|
| One row per link, sorted by kr/unit | Agent and UI tell the same story; "cheapest per unit" becomes answerable over MCP. | ✓ |
| Best link per store only | Keeps today's compact table, but hides that a bigger pack at the same store is the actual win. | |
| Leave MCP untouched | Smallest blast radius, but leaves the `priser` skill answering with a stale unit-price notion. | |

**User's choice:** One row per link, sorted by kr/unit → D-13
